## Context

Tachi's STRIDE + AI threat-modeling pipeline ships, post-F-5 (Feature 229 merged 2026-04-27), 9 AI-tier detection agents covering input-side attacker injection (`prompt-injection`, `data-poisoning`, `model-theft`, `agent-autonomy`, `tool-abuse`), downstream-execution-sanitization (`output-integrity` — LLM05:2025), factual-integrity (`misinformation` — LLM09:2025), agent-to-human communication-axis trust exploitation (`human-trust-exploitation` — ASI09:2026 communication axis), and infrastructure-availability + economic-damage extension on the LLM10:2025 surface via two-agent enrichment of `denial-of-service` (Cat 12/13) and `model-theft` (Cat 10/11) per F-5. The OWASP LLM Top 10:2025 framework rolls up to 10/10 Covered post-F-5; the OWASP Agentic Security Top 10:2026 framework rolls up to 10/10 Covered post-F-4 (combined OWASP AI top-10 = 20/20 Covered post-F-5). What remains silent on the OWASP Machine Learning Security Top 10:2023 framework is six entries: **ML01 Input Manipulation Attack, ML03 Model Inversion Attack, ML04 Membership Inference Attack, ML06 AI Supply Chain Attacks (both corpus-side and artifact-side facets), ML07 Transfer Learning Attack, and ML08 Model Skewing**. ML02, ML05, ML09, and ML10 are already Covered through F-1 (`output-integrity` for ML09) and pre-BLP-01 work on `data-poisoning` Cat 1-7 (for ML02 + ML10) and `model-theft` Cat 1-9 (for ML05).

BLP-01 (Better LLM Protection initiative, documented in `_internal/strategy/BLP-01.md`) identifies the OWASP ML Top 10:2023 coverage bundle as Tier 2 priority F-6, following Tier 1 closure across F-1 (Feature 201) `output-integrity`, F-2 (Feature 206) `misinformation`, F-3 (Feature 219) `tool-abuse` enrichment for ASI07:2026, F-4 (Feature 224) `human-trust-exploitation` for ASI09:2026 communication axis, and F-5 (Feature 229) two-agent enrichment for LLM10:2025. **F-6 is the third execution of the Heuristic A enrichment branch** (after F-3 single-agent scope and F-5 two-agent scope) and the **first at three-agent scope**: rather than authoring three new agents (`adversarial-input-manipulation`, `predictive-ml-poisoning`, `predictive-ml-extraction`) as 10th/11th/12th AI-tier siblings, F-6 enriches THREE existing host agents — `tampering` (with one new Pattern Category — Cat 10 Adversarial Input Manipulation against deployed predictive ML classifiers), `data-poisoning` (with three new Pattern Categories — Cat 8 Transfer Learning Supply Chain, Cat 9 Feedback-Loop Model Skewing, Cat 10 Predictive-ML Supply Chain Completeness), and `model-theft` (with three new Pattern Categories — Cat 12 Model Inversion, Cat 13 Membership Inference, Cat 14 Predictive-ML Artifact Supply Chain). The rationale — signal-class identity at three-agent scope, with explicit cross-agent vector decomposition for ML06 (two-facet corpus-side / artifact-side split) and explicit disjoint-architectural-tells decomposition for ML03 vs ML04 (input reconstruction vs training-set membership determination) on the same prediction-API surface — is operationalized below.

## Decision

We adopt the Heuristic A enrichment of the existing `tampering`, `data-poisoning`, and `model-theft` agents for OWASP ML Top 10:2023 closure across six entries (ML01 + ML03 + ML04 + ML06 corpus-side + ML06 artifact-side + ML07 + ML08) per the **10 numbered decisions** below. The feature ships with **zero new agent files**, **zero new skill directories**, **zero schema bumps**, **zero consumers-list edits**, **zero functional orchestrator edits**, and **zero `source_attribution` populator wiring extensions** — F-6 is the second-smallest-per-coverage-unit BLP-01 detection delivery to date (six ML Top 10 entries closed across seven Pattern Categories via three host enrichments, comparable to F-5's two LLM10 sub-classes via two host enrichments) and the **first at three-agent enrichment scope**. Six additive file edits across three host agents and three companion catalogs, one new ADR (this file under Proposed → Accepted), one new mutation-target architecture (`examples/predictive-ml-app/`), and zero functional dispatch edits constitute the entire delivery surface. The 22-file zero-edit invariant (post-F-5 inventory: 28 detection files; F-6 edits 6 host files; 22 unchanged) and the 6-baseline byte-identity gate are preserved by construction.

### Decision 1 — Heuristic A enrichment vs. new agent at THREE-AGENT scope — signal-class identity rationale

We enrich THREE existing host agents — `tampering` (Cat 10), `data-poisoning` (Cat 8 + 9 + 10), and `model-theft` (Cat 12 + 13 + 14) — rather than authoring three new agents (`adversarial-input-manipulation`, `predictive-ml-poisoning`, `predictive-ml-extraction`) as 10th/11th/12th AI-tier siblings. The rationale rests on **signal-class identity at three-agent scope** — each ML Top 10 sub-pattern naturally cleaves to the existing host agent whose pre-existing detection vocabulary and mitigation taxonomy best match the architectural-tell:

| ML Top 10 sub-pattern | Owning agent / category | Pre-existing host signal class (rationale) |
|---|---|---|
| Adversarial input manipulation against deployed predictive ML inference (ML01) | `tampering` Cat 10 | tampering already covers integrity-tier attacks against Process input boundaries (Cat 1-9: input injection, deserialization, supply chain integrity gaps). Cat 10 extends to predictive-ML-tier integrity attacks at the inference-input boundary — same signal class (untrusted-input → output deviation) at a different abstraction layer (predictive-ML-tier vs. generic-application-tier). |
| Transfer learning supply chain — pretrained weight integrity (ML07) | `data-poisoning` Cat 8 | data-poisoning already covers training-data integrity attacks against Data Stores and Data Flows (Cat 1-7: training data manipulation, RAG poisoning, fine-tuning supply chain, backdoor triggers). Cat 8 extends to pretrained-weight supply-chain integrity at the fine-tuning step — same signal class (training-pipeline integrity) at a different abstraction layer (pretrained-weight artifact vs. training-data corpus). |
| Feedback-loop active-learning skewing (ML08) | `data-poisoning` Cat 9 | data-poisoning already covers training-data integrity. Cat 9 extends to active-learning feedback loops where production predictions feed back into training — same signal class (training-pipeline integrity) at a different temporal layer (online learning vs. batch training). |
| Predictive-ML supply chain — corpus side (ML06 corpus facet) | `data-poisoning` Cat 10 | data-poisoning already covers RAG / training-data supply chain. Cat 10 extends to predictive-ML dataset repositories + feature stores + MLOps registry corpus-side — same signal class (corpus-side supply-chain integrity) for predictive-ML training corpora rather than LLM training corpora. |
| Model inversion attack (ML03) | `model-theft` Cat 12 | model-theft already covers extraction attacks against deployed models (Cat 1-9: weight exfiltration, API extraction, side-channel reconstruction, fine-tuned model theft, supply chain compromise, exfiltration via inference API). Cat 12 extends to gradient-inversion + black-box optimization attacks reconstructing training-set inputs — same signal class (extraction via inference) at a different attack-target layer (training-set inputs vs. model weights). |
| Membership inference attack (ML04) | `model-theft` Cat 13 | model-theft already covers extraction. Cat 13 extends to confidence-thresholding + shadow-model attacks determining training-set membership — same signal class (extraction via inference) at a different attack-target layer (membership determination vs. weight/input reconstruction). Shares MITRE ATLAS AML.T0024 catalog reference with Cat 12 but disjoint architectural-tells per Decision 5. |
| Predictive-ML supply chain — artifact side (ML06 artifact facet) | `model-theft` Cat 14 | model-theft already covers supply chain compromise (Cat 7: weights / checkpoints / model artifacts). Cat 14 extends to predictive-ML model registry + weight-artifact-storage + serving-time integrity — same signal class (model-artifact supply-chain integrity) for predictive-ML artifacts rather than LLM artifacts. |

**Heuristic A signal-class discipline test**: are F-6 sub-patterns more naturally adjacent to existing host-agent Pattern Categories or to three new agents (separate agent files, separate trigger keyword sets, separate dispatch paths)? Architecture answer: ADJACENT to existing host-agent categories AT THREE-AGENT SCOPE. The seven new Pattern Categories naturally cleave by signal class (integrity attack against input / training-pipeline integrity / extraction via inference) — and that cleave aligns precisely with the existing `tampering` (integrity) vs. `data-poisoning` (training-pipeline) vs. `model-theft` (extraction) signal-class boundary. Splitting F-6 into three new agents would (a) duplicate trigger keywords across tampering + data-poisoning + model-theft + 3 new agents, (b) fragment the ML Top 10 attack surface across six artificial agent boundaries, (c) force ID-prefix segmentation across what is structurally three emission streams (`T-{N}` for input-integrity surface + `D-{N}` for training-pipeline surface + `LLM-{N}` for extraction surface), and (d) inflate AI-tier agent count from 9 to 12 without producing detection-vocabulary or mitigation-taxonomy benefits over enrichment.

**Authority acknowledgment — inheritance, not re-adjudication**: ADR-030 Decision 1 established the **signal-class taxonomy in the LLM tier** (input-side / downstream-execution-sanitization / factual-integrity / psychology-linguistics — three-way partition formalized at F-1; F-2 inherited the partition for factual-integrity). ADR-032 (F-3) demonstrated **enrichment-branch as a viable alternative to new-agent authoring** at single-agent scope. ADR-034 (F-5) extended the precedent to **two-agent scope with explicit cross-agent vector decomposition** (Q1 SPLIT). **F-6 demonstrates the same signal-class discipline applied across three host agents simultaneously** — the ML Top 10 surface decomposes naturally into three signal classes (integrity / training-pipeline / extraction) and each maps to an existing host agent. This is **inheritance of the rule** at three-agent scope, not re-adjudication. SDR-001 Decision 4 is the locked resolution at the BLP-01 strategy tier; this Decision 1 is the operational realization at the F-6 architectural tier. ADR-034 lines 192-204 forecast that F-6 / F-7 Tier 2 ML+Mobile bundles may invoke the enrichment branch at multi-agent scope depending on signal-class analysis at PRD time — F-6 fulfills the multi-agent forecast at three-agent scope.

**Forward scope marker**: F-7 Tier 2 OWASP Mobile Top 10 bundle at planned 5-agent fan-out may invoke the enrichment branch at single-agent or multi-agent scope depending on signal-class analysis at PRD time; F-6 establishes three-agent enrichment as a viable operational pattern beyond F-3 single-agent and F-5 two-agent precedents, with extensibility to ≥4-agent fan-out implied by structural symmetry.

### Decision 4 — ML06 two-facet disjoint architectural-tells (corpus-side data-poisoning Cat 10 vs artifact-side model-theft Cat 14)

ML06 (AI Supply Chain Attacks) decomposes across two owning categories with disjoint architectural-tells:

- **Corpus-side facet → data-poisoning Cat 10** owns dataset repositories (e.g., public Kaggle dataset, internal merchant transaction history), feature stores (e.g., Feast / Tecton), MLOps registry corpus-side (e.g., MLflow ingest path), and the training-data path (training pipeline → feature store → fine-tuning input). Architectural-tell: training-data ingestion or feature-store write path without IAM-enforced write-audit, dataset-checksum manifest, integrity verification, or model-card review gate.
- **Artifact-side facet → model-theft Cat 14** owns the model registry (e.g., MLflow artifact-promotion path, SageMaker Model Registry, Vertex AI Model Registry), weight-artifact storage (e.g., S3 bucket holding fine-tuned model checkpoints, Hugging Face private hub), and serving-time integrity (e.g., model-load-time checksum verification on the prediction API process). Architectural-tell: model-registry promotion or weight-artifact-storage write path without signed-artifact policy (Sigstore-style or KMS-backed cryptographic attestation), without registry IAM with promotion-gate review, without integrity verification at model-load time, or without immutable artifact storage with audit logging.

**Same architecture surfaces both** — neither is a duplicate. The fictional `examples/predictive-ml-app/` mutation target exhibits both signals (Public Dataset Repository + Internal Merchant Transaction History + Feast Feature Store on the corpus side; MLflow Model Registry + Weight Checkpoint Storage on the artifact side; HuggingFace Hub Pretrained-Weights Registry contributing to both via the Fine-Tuning Service Process). Architect formalizes the disjoint-tells decomposition here at the ADR layer; the inline Pattern Category Disambiguation subsections (per Decision 9) on the data-poisoning and model-theft companions formalize the inline narrative for adopter readers.

**Why a two-facet split rather than single-owner**: the ML06 surface is structurally bifurcated by the model-training timeline. Corpus-side gaps (ML06 corpus) propagate at training time (poisoned corpus → poisoned model → poisoned predictions). Artifact-side gaps (ML06 artifact) propagate at serving time (poisoned artifact → poisoned predictions even with clean training corpus). The two facets have disjoint mitigation taxonomies — corpus-side mitigations are dataset-checksum manifests + IAM-enforced write-audit + model-card review gates; artifact-side mitigations are signed-artifact policies + registry IAM with promotion-gate review + integrity verification at model-load time. Forcing a single owner would either understate the artifact-side mitigations (if data-poisoning owns it) or understate the corpus-side mitigations (if model-theft owns it). Splitting by owning agent preserves the existing signal-class boundary (data-poisoning = training-pipeline integrity; model-theft = artifact integrity) and makes the mitigation taxonomy on each side of the split distinct and complete.

### Decision 5 — ML03 vs ML04 disjoint architectural-tells on shared prediction-API surface (Cat 12 input reconstruction vs Cat 13 membership determination)

ML03 (Model Inversion) and ML04 (Membership Inference) both target a deployed prediction-API endpoint and both share the catalog-resolvable `MITRE ATLAS AML.T0024 — Exfiltration via ML Inference API` reference. **They are NOT duplicates** — disjoint architectural-tells per the decomposition below:

- **Cat 12 (Model Inversion) — input reconstruction**: white-box gradient inversion (attacker has model weight access; inverts prediction outputs to reconstruct training-set inputs) + black-box optimization for input reconstruction (attacker queries prediction API with crafted inputs and uses the confidence outputs to reconstruct the training-set inputs that maximize the model's predictions for a target class). Architectural-tell: prediction API serving classifier with sensitive training data (medical-imaging classifier, financial-transaction classifier, personal-data classifier) AND DP-SGD on training absent AND output-perturbation noise injection absent AND query-rate throttling per tenant absent AND model-extraction-pattern detection absent.
- **Cat 13 (Membership Inference) — training-set membership determination**: confidence-thresholding (attacker queries prediction API with candidate training inputs and uses output confidence to determine whether the input was a training-set member) + shadow-model attacks (attacker trains shadow models on similar data distributions and uses comparative confidence patterns to determine training-set membership in the target model). Architectural-tell: prediction API returning confidence values AND shadow-model attack feasibility (training-data distribution publicly known or guessable) AND label-only response mode missing AND DP-SGD absent AND confidence-output truncation absent AND training-data minimization not enforced.

**Both findings emit on the same architectural footprint** — a prediction API serving a classifier on sensitive training data without DP-SGD and without per-tenant throttling. The disjoint architectural-tells make the mitigation narratives on each finding non-redundant: Cat 12 mitigation emphasizes DP-SGD + output-perturbation noise + query-rate throttling + extraction-pattern detection; Cat 13 mitigation emphasizes DP-SGD + label-only response mode + confidence-output truncation + training-data minimization. The Cat 12 mitigation covers input-reconstruction-side defenses; the Cat 13 mitigation covers membership-determination-side defenses. **Shared `MITRE ATLAS AML.T0024` reference is appropriate** — both attack classes exfiltrate via the inference API — but the architectural-tell decomposition prevents single-finding consolidation and ensures the adopter receives complete mitigation guidance for both attack classes.

**Why a two-category split rather than single-category**: model-theft Cat 6 (Unbounded Inference Consumption — pre-existing) covers per-tenant quota / cost-control / billing-attribution gaps at the abstraction level. F-5's Cat 10 + Cat 11 covered the LLM10:2025 specific attack vectors below the abstraction level Cat 6 covers. F-6's Cat 12 + Cat 13 covers the predictive-ML extraction attack vectors below the abstraction level Cat 6 / Cat 7 / Cat 8 cover (which are LLM-tier extraction). The two-category split for predictive-ML extraction maintains the same abstraction-layer discipline as F-5: the new categories are below the abstraction level the pre-existing categories cover, and each new category covers a distinct attack vector with disjoint mitigation taxonomy.

The pre-naming makes the deferral path safe enough to attempt the full 7-category ship.

**Rejected as default**: ship full 7-category bundle. **Accepted as R5 contingency**: if Heuristic A three-agent emergent issues materialize during build, defer the pre-named ML06 pair (data-poisoning Cat 10 + model-theft Cat 14) per spec OoS-15.
