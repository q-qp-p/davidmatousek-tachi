## Context

Tachi's STRIDE + AI threat-modeling pipeline ships, post-F-6 (Feature 232 merged 2026-04-28), 12 AI-tier detection agents and 5 STRIDE-tier detection agents — totaling **28 detection-tier files** in inventory (14 host agents + 14 companion `detection-patterns.md` reference files; the AI-tier rolled to 12 because pre-BLP-01 inventory + F-1 `output-integrity` + F-2 `misinformation` + F-4 `human-trust-exploitation` net-new agents added three; the STRIDE-tier remained at 5 — `spoofing`, `tampering`, `repudiation`, `info-disclosure`, `privilege-escalation`, plus `denial-of-service` from the earlier inventory). The OWASP three-framework total rolled to **30/30 Covered** post-F-6: LLM Top 10:2025 = 10/10 (closed at F-5), Agentic Top 10:2026 = 10/10 (closed at F-4), ML Top 10:2023 = 10/10 (closed at F-6). What remains silent on the **OWASP Mobile Top 10:2024** framework is all 10 entries: **M1 Improper Credential Usage, M2 Inadequate Supply Chain Security, M3 Insecure Authentication/Authorization, M4 Insufficient Input/Output Validation, M5 Insecure Communication, M6 Inadequate Privacy Controls, M7 Insufficient Binary Protections, M8 Security Misconfiguration, M9 Insecure Data Storage, M10 Insufficient Cryptography**.

## Decision

We adopt the following 10 numbered decisions to deliver F-7 OWASP Mobile Top 10:2024 closure across up to five existing STRIDE host agents at four-or-five-agent enrichment scope, with M8 dual-host carve and M4 cross-agent boundary as additional structural complexity beyond ADR-035 three-agent precedent.

### Decision 1 — Heuristic A enrichment at four-or-five-agent scope (NOT a new mobile-platform agent)

We enrich 5 existing STRIDE host agents in the dual-host path (`spoofing` + `tampering` + `info-disclosure` + `privilege-escalation` + `repudiation`) or 4 in the single-host fallback (where M8 is consolidated into either `privilege-escalation` or `repudiation` alone) rather than authoring a new `mobile-platform` host agent as a 13th AI/STRIDE-tier sibling.

**Rejected alternative**: a new `mobile-platform` host agent with 10 Pattern Categories (one per M-item) consolidating M1-M10 in a single agent file. Rejected because (a) it would fragment the M1-M10 attack surface across 11 locations of the codebase (the new mobile-platform agent + 5 STRIDE companions which would still need cross-references to maintain dispatch coherence), (b) it would violate the Heuristic A signal-class consolidation rule (M-items naturally cleave by signal class — identity impersonation / data integrity / confidentiality leakage / privilege gain / accountability loss — and that cleave aligns precisely with the existing STRIDE host-agent boundary), (c) it would duplicate trigger keywords across spoofing + tampering + info-disclosure + privilege-escalation + repudiation + 1 new agent, (d) it would force ID-prefix segmentation across what is structurally five emission streams (`S-{N}` for identity-impersonation surface + `T-{N}` for data-integrity surface + `I-{N}` for confidentiality-leakage surface + `E-{N}` for privilege-gain surface + `R-{N}` for accountability-loss surface), and (e) it would inflate detection-tier agent count from 14 to 15 without producing detection-vocabulary or mitigation-taxonomy benefits over enrichment.

**Heuristic A signal-class discipline test at four-or-five-agent scope**: are F-7 sub-patterns more naturally adjacent to existing host-agent Pattern Categories or to a new mobile-platform agent? Architecture answer: ADJACENT to existing STRIDE host-agent categories. The 10 new Pattern Categories naturally cleave by signal class — identity impersonation (M1 mobile credential usage; M3 mobile authn/authz) → `spoofing`; data integrity (M2 mobile supply chain; M4 mobile IPC input; M7 mobile binary protections) → `tampering`; confidentiality leakage (M5 insecure mobile communication; M6 inadequate mobile privacy; M9 insecure mobile data storage; M10 insufficient mobile cryptography) → `info-disclosure`; privilege gain (M8 mobile misconfiguration enabling debug endpoints + ContentProvider exports + missing app-attestation) → `privilege-escalation`; accountability loss (M8 mobile misconfiguration enabling missing audit logging + disabled crash reporting + debug log leakage) → `repudiation`. The cleave aligns precisely with the existing five STRIDE signal-class boundaries.

**Splitting F-7 into a new mobile-platform agent would**:
1. Duplicate trigger keywords across `spoofing` + `tampering` + `info-disclosure` + `privilege-escalation` + `repudiation` + 1 new `mobile-platform` agent (the trigger keyword "mobile" / "Android" / "iOS" / "deep-link" / "Intent" / "ContentProvider" would have to live in 6 places to maintain dispatch correctness on hybrid mobile+web architectures).
2. Fragment the Mobile Top 10 attack surface across **eleven artificial agent boundaries** (the new mobile-platform agent plus 5 STRIDE companions which would still need cross-references — the new agent cannot stand alone because mobile attacks fundamentally engage STRIDE-tier signal classes).
3. Force ID-prefix segmentation across what is structurally **five emission streams** (`S-{N}` for identity-impersonation surface + `T-{N}` for data-integrity surface + `I-{N}` for confidentiality-leakage surface + `E-{N}` for privilege-gain surface + `R-{N}` for accountability-loss surface) into a single artificial `MOB-{N}` namespace that would erase the signal-class differentiation that downstream consumers (`risk-scorer`, `control-analyzer`, `threat-report`) rely on.
4. Inflate detection-tier agent count from 14 to 15 without producing detection-vocabulary or mitigation-taxonomy benefits over enrichment — the mobile-specific detection vocabulary (Intent extras, ContentProvider gates, biometric APIs, FLAG_SECURE, allowBackup, KDF strength on PINs) already maps cleanly to the existing STRIDE signal classes; a new agent would re-bundle this vocabulary into a non-orthogonal cross-cut that complicates rather than clarifies dispatch.

**Authority acknowledgment — inheritance, not re-adjudication**: ADR-030 Decision 1 established the **signal-class taxonomy** in the LLM tier; ADR-032 (F-3) demonstrated **enrichment-branch as a viable alternative to new-agent authoring** at single-agent scope; ADR-034 (F-5) extended the precedent to **two-agent scope with explicit cross-agent vector decomposition** (Q1 SPLIT); ADR-035 (F-6) extended the precedent to **three-agent scope with explicit ML06 two-facet split + ML03/ML04 disjoint-tells**. **F-7 demonstrates the same signal-class discipline applied across four or five host agents simultaneously** — the OWASP Mobile Top 10:2024 surface decomposes naturally into five signal classes (identity impersonation / data integrity / confidentiality leakage / privilege gain / accountability loss) and each maps to an existing STRIDE host agent. This is **inheritance of the rule** at four-or-five-agent scope, not re-adjudication. SDR-001 Decision 4 is the locked resolution at the BLP-01 strategy tier; this Decision 1 is the operational realization at the F-7 architectural tier. The ADR-035 closing forward-scope marker (line 77 verbatim, cited in Decision 6 below) explicitly forecast that F-7 four-or-five-agent enrichment was viable through structural symmetry — F-7 fulfills the forecast.

**Heuristic A scope progression across BLP-01**:

| Feature | Heuristic A Scope | Host Agents | OWASP Entries Covered |
|---------|-------------------|-------------|------------------------|
| F-3 (Feature 219) | single-agent | 1 (`tool-abuse`) | ASI07:2026 (1 entry) |
| F-5 (Feature 229) | two-agent | 2 (`denial-of-service`, `model-theft`) | LLM10:2025 (1 entry, Q1 SPLIT vector decomposition) |
| F-6 (Feature 232) | three-agent | 3 (`tampering`, `data-poisoning`, `model-theft`) | ML01/ML03/ML04/ML06/ML07/ML08:2023 (6 entries, 2-facet split on ML06 + disjoint-tells on ML03/ML04) |

The progression doubles the host count and quadruples the OWASP-entry coverage from F-6 to F-7 (3 hosts × 6 entries → 5 hosts × 10 entries). The agent-count growth is bounded by the size of the existing STRIDE roster (5 STRIDE host agents); Mobile Top 10 happens to engage all 5 STRIDE signal classes simultaneously, which is the maximum scope reachable through enrichment alone without authoring new STRIDE-tier signal classes.

### Decision 2 — Additive-only edit discipline per ADR-023 Decision 3 at four-or-five-agent scope

We follow ADR-023 Decision 3's **additive-only edit discipline** at four-or-five-agent scope. F-7 is the fourth BLP-01 feature (after F-3 single-agent, F-5 two-agent, F-6 three-agent) where additive-only edits to existing files are the **entire** delivery surface, and the **first at four-or-five-agent scope**.

Specifically, across the 10 host files in the dual-host path (or 8 in the single-host fallback), edits are append-only — pre-existing prose stays byte-identical pre/post (grep-checkable). Three additive anchor points per host agent:

(a) **Metadata `owasp_references` extension** — the existing list of OWASP cross-references in each host agent's frontmatter gains the appropriate M-item entries (e.g., `owasp_references += [OWASP M1:2024, OWASP M3:2024]` for spoofing). Pre-existing entries unchanged.

(b) **`## Purpose` 1-3 line append** — the existing Purpose section gains a 1-3 line addition naming the mobile-tier surface the host agent now covers (e.g., for spoofing: a one-sentence addition naming "mobile credential storage and mobile session handling on Android/iOS surfaces"). Pre-existing Purpose prose unchanged.

(c) **Detection Workflow Step 5 references list extension** — the existing references list at Step 5 gains the appropriate OWASP Mobile catalog citations (e.g., `OWASP M1:2024, OWASP M3:2024` for spoofing; with MITRE ATT&CK Mobile T1474/T1626/T1398 cited prose-only per Decision 7 catalog-absence rule). Pre-existing references unchanged.

Companion `detection-patterns.md` files receive Pattern Category appends + Pattern Category Disambiguation subsection (Decision 9) + Primary Sources extension. Pre-existing Pattern Categories 1-N are preserved byte-identically.

**Byte-identity test contract per Wave 2.x execution**: prior to staging any companion-catalog edits, the build agent extracts each pre-existing Pattern Category (Cat 1 through Cat N) from the post-F-6 baseline using line-range slicing into a per-category byte buffer; after the edits, the same line-range slicing is re-applied and the buffers are compared byte-for-byte. A non-byte-identical diff on any pre-existing Cat is a Wave 2.x build BLOCKER per the additive-only edit discipline. This is structurally symmetric with ADR-035's T038/T044 byte-identity tests (which verified Cat 1-9 of tampering, Cat 1-7 of data-poisoning, Cat 1-11 of model-theft post-F-5 inventory). F-7's byte-identity test surface expands to include Cat 1-N of all five enriched companions (or four in single-host fallback).

**Anchor point ordering across host file edits**: anchor (a) → anchor (b) → anchor (c) within each host agent file. ### Decision 4 — M8 dual-host disjoint architectural-tells (privilege-gain variant vs accountability-loss variant)

We carve M8 (Security Misconfiguration) across **two owning STRIDE host agents** with **disjoint architectural-tells**:

**M8 in `privilege-escalation`** owns the **privilege-gain variant**: mobile-misconfiguration enabling privilege gain. Architectural tells: exposed debug endpoints (e.g., `/debug` on a release build, accessible debug ContentProvider), default permissive ContentProvider/Service exports (`android:exported="true"` without permission gate), missing app-attestation (no Play Integrity / DeviceCheck on security-critical operations), missing root-detection on security-critical features, default-permissive intent filters allowing untrusted callers to invoke privileged components.

**M8 in `repudiation`** owns the **accountability-loss variant**: mobile-misconfiguration enabling accountability loss. Architectural tells: missing audit logging on security-relevant events (auth state transitions, biometric prompts, sensitive transactions), disabled crash reporting in production (Firebase Crashlytics / Sentry disabled or scrubbed too aggressively), debug logs leaking sensitive data via `Log.d` / `NSLog` in release builds, missing tamper-evident timestamping on transaction logs, audit log writers with no integrity protection.

**Same architecture exhibiting BOTH variants may legitimately surface both findings without duplication** — the architectural-tells are disjoint and the mitigations are different (privilege-escalation mitigations target authorization gating; repudiation mitigations target audit-trail completeness).

This mirrors **ADR-035 D-4 ML06 two-facet precedent** (corpus-side data-poisoning vs artifact-side model-theft) — F-7 carries forward the two-facet pattern at the **M8 architectural-tell layer** rather than at the OWASP-entry layer (i.e., one OWASP entry M8 carved across two host agents at the architectural-tell axis, not at the entry-decomposition axis like ML06's corpus-vs-artifact carve which decomposed one OWASP entry into two distinct attack-surface facets). The dual-host carve is structurally analogous but operates at a different axis layer.

**Structural-axis comparison**:

| Feature | OWASP Entry | Carve Axis | Owning Agents |
|---------|-------------|------------|---------------|
| F-6 (ADR-035) | ML06 (AI Supply Chain Attacks) | Entry-decomposition (corpus-vs-artifact) | data-poisoning Cat 10 + model-theft Cat 14 |
| F-7 (this ADR) | M8 (Security Misconfiguration) | Architectural-tell (privilege-gain vs accountability-loss) | privilege-escalation + repudiation |

The F-6 axis decomposes ONE OWASP entry into TWO distinct attack-surface facets (corpus and artifact are both genuine attack-surface targets). The F-7 axis decomposes ONE OWASP entry across TWO complementary architectural-tells (the same misconfiguration architecture exhibits BOTH variants — privilege gain and accountability loss are interlocking consequences of misconfiguration, not distinct attack-surface targets). Both carves preserve **disjoint mitigation vocabularies**: F-6's ML06-corpus mitigations are dataset-integrity-oriented while ML06-artifact mitigations are registry-integrity-oriented; F-7's M8-privilege-gain mitigations are authorization-gating-oriented (attestation + permission gates + root-detection) while M8-accountability-loss mitigations are audit-trail-oriented (audit logging + crash reporting + tamper-evident timestamping).

**Single-host fallback**: if Triad governance at Wave 1.0 surfaces a strong preference to consolidate M8 in either `privilege-escalation` alone (favoring the privilege-gain reading) or `repudiation` alone (favoring the accountability-loss reading), the canonical mapping table reduces to 9 closure rows + 1 reference row and the host count reduces from 5 to 4. The dual-host path is the architect-recommended default per the disjoint-architectural-tells argument above; the single-host fallback is documented for completeness. **The architect recommends the dual-host path** because (a) M8 in OWASP Mobile Top 10:2024 explicitly enumerates BOTH attack vectors as independent risks (debug-endpoint exposure on the privilege-gain side; missing audit logging on the accountability-loss side); (b) consolidating into a single host would force an arbitrary primary-axis choice that downstream consumers would have to reverse-engineer; (c) the disjoint mitigation vocabularies argue against consolidation because a single host would have to carry both vocabularies under one Pattern Category, fragmenting the signal-class identity test that motivates Decision 1.

**Architectural-tell glossary for M8 dual-host carve** (Wave 2.x companion-catalog authoring reference):

| Architectural Tell | Owning Variant | Example Mitigation |
|---------------------|----------------|---------------------|
| Exposed debug endpoint on release build (`/debug` HTTP route, debug ContentProvider with `android:exported="true"`) | privilege-gain | Strip debug routes / providers from release builds via build-flavor configuration |
| Default permissive ContentProvider/Service exports without permission gate | privilege-gain | Set `android:exported="false"` or add `android:permission` attribute |
| Missing app-attestation on security-critical operations (no Play Integrity / DeviceCheck) | privilege-gain | Integrate Play Integrity (Android) / DeviceCheck (iOS) on transaction-confirmation paths |
| Missing root-detection on security-critical features | privilege-gain | Integrate SafetyNet / iOS jailbreak detection; degrade UX or block features on rooted devices |
| Default-permissive intent filters allowing untrusted callers | privilege-gain | Add `android:exported="false"` for non-public intent receivers |
| Missing audit logging on auth state transitions | accountability-loss | Emit audit log entries on every login / logout / step-up / token-refresh event |
| Missing audit logging on biometric prompt outcomes | accountability-loss | Emit audit log entries on biometric prompt success / failure / cancellation |
| Disabled crash reporting in production (Crashlytics / Sentry off) | accountability-loss | Enable crash reporting; configure PII scrubbing without disabling the channel |
| Debug logs leaking sensitive data via `Log.d` / `NSLog` in release | accountability-loss | Replace with `Log.i` / `OSLog` with explicit redaction; gate on build flavor |

The glossary is referenced by Wave 2.x companion-catalog authors when populating Pattern Categories in `tachi-privilege-escalation/references/detection-patterns.md` and `tachi-repudiation/references/detection-patterns.md`. The disjoint mitigation columns (different mitigation vocabularies) operationalize the disjoint-architectural-tells argument from Decision 4 above.

### Decision 5 — M4 cross-agent boundary clarification with F-1 `output-integrity`

We carve M4 (Insufficient Input/Output Validation) at the **mobile-IPC-input vs LLM-output-sanitization boundary** between F-7's tampering-Cat-12 and F-1's `output-integrity` agent.

**Cat 12 Mobile IPC Input Validation (tampering host — F-7)** owns mobile-IPC-input-side validation: deep-link parameters (e.g., `myapp://transfer?amount=...&payee=...`), Intent extras (`Intent.getStringExtra("...")`), URL-scheme parameters (iOS custom URL schemes), exported ContentProvider gates (untrusted callers passing query selectors), pasteboard injection paths (clipboard content flowing to action-builders), share-sheet payloads (Receive Intent payload validation).

**F-1's `output-integrity` agent (LLM05:2025)** owns LLM-output-side sanitization: LLM-generated content flowing into browser sinks (e.g., HTML rendering of LLM output without escaping), SQL sinks (LLM-emitted SQL fragments), shell sinks (LLM-emitted shell commands), and other downstream-execution targets.

**Disjoint architectural-tells prevent duplicate emission on hybrid LLM+mobile architectures**: a mobile app integrating an on-device LLM with a hybrid WebView surface may legitimately surface BOTH a Cat 12 finding (mobile IPC validation gap on the deep-link-to-LLM-prompt path) AND an OI finding (LLM-output-to-WebView sink without sanitization), without duplication — the input axis (Cat 12) and the output axis (OI) are architecturally orthogonal.

**Worked example — on-device LLM banking assistant**: a fintech mobile app embedding an on-device LLM as a transaction-confirmation assistant might receive a deep-link Intent (`myapp://confirm?txid=...&prompt=...`) where the `prompt` extra is forwarded to the LLM without validation, and the LLM's output is then rendered into a hybrid WebView confirmation pane. This architecture has TWO orthogonal vulnerabilities:
1. **Mobile IPC input axis (Cat 12 — tampering)**: the `prompt` extra in the deep-link Intent is unvalidated; an attacker may craft a malicious Intent (e.g., via a phishing app on the same device) that injects a hostile prompt for the LLM. Mitigation: validate / sanitize Intent extras before forwarding; signed deep-link payloads; user-mediated confirmation step for prompt forwarding.
2. **LLM output sanitization axis (OI — output-integrity)**: the LLM output rendered into the WebView is unsanitized; an injected prompt may produce HTML/JavaScript output that executes in the WebView context. Mitigation: sanitize LLM output before WebView injection; CSP enforcement on the WebView; output-shape validation.

Both findings emit on the same architecture without duplication. The mitigations are different (input gating vs output sanitization), the OWASP citations are different (M4:2024 vs LLM05:2025), and the hosts are different (tampering vs output-integrity).

This mirrors **ADR-035 D-5 ML03-vs-ML04 disjoint-tells precedent** (shared AML.T0024 catalog reference but disjoint mitigation vocabularies) at F-7 cross-axis layer (mobile IPC input vs LLM output sanitization — disjoint architectural-tells, NOT shared catalog: M4 OWASP citation lives entirely in `tampering`, LLM05:2025 citation lives entirely in `output-integrity`; the cross-agent boundary is at the architectural-tell axis without catalog overlap). **Asymmetry with ADR-035 D-5**: ADR-035 D-5 carved across two ML-tier OWASP entries that SHARE a MITRE ATLAS catalog reference (AML.T0024) — the cross-agent boundary preserved disjoint mitigation vocabularies despite shared catalog. ADR-036 D-5 carves across one Mobile-tier OWASP entry (M4) and one LLM-tier OWASP entry (LLM05) with NO shared catalog — the cross-agent boundary is purely at the architectural-tell axis. F-7's D-5 is therefore a **structural extension** of ADR-035's D-5 to non-shared-catalog cross-axis decompositions.

### Decision 6 — No schema bump (fourth no-bump enrichment after F-3/F-5/F-6; first at four-or-five-agent scope)

F-7 does not bump `schemas/finding.yaml`. The existing `id.pattern` regex already enumerates all five STRIDE prefixes F-7 emits into:

```
^(S|T|R|I|D|E|AG|LLM|AGP|OI|MI|TE)-\d+$
```

F-7 emits `S-{N}` (spoofing host — M1, M3 findings), `T-{N}` (tampering host — M2, M4, M7 findings), `I-{N}` (info-disclosure host — M5, M6, M9, M10 findings), `E-{N}` (privilege-escalation host — M8 privilege-gain findings), and `R-{N}` (repudiation host — M8 accountability-loss findings). **No new prefix is introduced**.

**Asymmetry with F-1, F-2, F-4**: F-1 (Feature 201) bumped `schema_version` 1.5 → 1.6 to add the `OI` prefix; F-2 (Feature 206) bumped 1.6 → 1.7 to add the `MI` prefix; F-4 (Feature 224) bumped 1.7 → 1.8 to add the `TE` prefix. **F-3 (Feature 219), F-5 (Feature 229), F-6 (Feature 232), and now F-7 (Feature 237) do NOT bump the schema** — F-7 is the fourth BLP-01 detection feature with no schema bump and the **first at four-or-five-agent scope**. The asymmetry is a structural consequence of Heuristic A enrichment vs. new-agent authoring: enrichment reuses existing prefixes by construction; new-agent authoring requires a new prefix and therefore a schema bump under ADR-031 Decision 8.

**ADR-031 Decision 8 — explicit asymmetry, NOT invocation**: ADR-031 Decision 8 established the regex-alternation minor-bump rule (any new ID prefix added to `id.pattern` regex requires a minor schema-version bump because the pattern is part of the contract, even though the change is additive-compatible). F-7 does NOT invoke this rule because F-7 introduces no new ID prefix. The asymmetry is recorded explicitly here so future BLP-01 features (F-8 Tier 3 web/API refinement bundle and beyond) can identify whether they fall in the schema-bump path (new agent / new prefix) or the no-schema-bump path (enrichment / existing prefix) at PRD time.

**Schema-bump-path vs no-bump-path decision tree** (codified across the BLP-01 ADR family for forward use at F-8+):

```
Does the new feature introduce a new ID prefix?
├── YES → schema-bump path (minor version bump per ADR-031 D8)
│        Examples: F-1 (OI prefix, 1.5→1.6), F-2 (MI prefix, 1.6→1.7), F-4 (TE prefix, 1.7→1.8)
│        Required: net-new agent file authoring + companion catalog + consumers-list edit
└── NO  → no-bump path (existing prefix reuse)
         Examples: F-3 (AG reuse, single-agent enrichment), F-5 (D + LLM reuse, two-agent),
                   F-6 (T + D + LLM reuse, three-agent), F-7 (S + T + I + E + R reuse, 4-or-5 agent)
         Required: additive-only edits to existing host agents per ADR-023 D3
```

The schema-bump path is appropriate when the new feature requires a new signal-class identity (e.g., F-1's `output-integrity` for downstream-execution-sanitization, F-2's `misinformation` for factual-integrity, F-4's `human-trust-exploitation` for communication-axis trust). The no-bump path is appropriate when the new feature extends an existing signal-class identity to a new attack surface (e.g., F-3's ASI07 inter-agent communication extending `tool-abuse`'s single-agent surface; F-5's LLM10 unbounded consumption extending `denial-of-service` and `model-theft`; F-6's ML Top 10 extending `tampering`/`data-poisoning`/`model-theft` to predictive-ML; F-7's Mobile Top 10 extending the five STRIDE host agents to mobile-platform tells). The decision tree is grounded in Heuristic A signal-class identity inheritance from ADR-030 D1.

**ADR-035 closing forward-scope marker — fulfilled at four-or-five-agent scope**: ADR-035 (F-6) closed with the following forward-scope marker (line 77 verbatim):

> **Forward scope marker**: F-7 Tier 2 OWASP Mobile Top 10 bundle at planned 5-agent fan-out may invoke the enrichment branch at single-agent or multi-agent scope depending on signal-class analysis at PRD time; F-6 establishes three-agent enrichment as a viable operational pattern beyond F-3 single-agent and F-5 two-agent precedents, with extensibility to ≥4-agent fan-out implied by structural symmetry.

F-7 fulfills this forecast: **F-7 invokes the multi-agent enrichment branch at four-or-five-agent scope** (5 agents in the dual-host M8 path; 4 in the single-host fallback). The signal-class analysis at PRD time identified five disjoint signal classes — identity impersonation / data integrity / confidentiality leakage / privilege gain / accountability loss — each mapping to exactly one existing STRIDE host agent. The dual-host M8 carve (Decision 4) operationalizes the "privilege gain" + "accountability loss" cleave as two distinct architectural-tell axes within a single OWASP entry. ADR-035's forecast of "extensibility to ≥4-agent fan-out implied by structural symmetry" is now operationally validated.

ADR-035 also forecasts at line 157 ("F-6 fulfills both forecasts at three-agent scope") that the chain ADR-032 → ADR-034 → ADR-035 → F-7 should preserve the no-schema-bump discipline if existing prefixes accommodate the new findings. Establishes precedent that **enrichment-branch fan-out is bounded by agent count, not schema surface** — the schema surface (regex alternation) bounds new-prefix creation, while the agent-count surface bounds enrichment scope; the two are decoupled.

### Decision 9 — Pattern Category Disambiguation requirement on 5 companions in dual-host path (4 in single-host fallback)

Each of the 5 enriched companions in the dual-host path (`tachi-spoofing/references/detection-patterns.md`, `tachi-tampering/`, `tachi-info-disclosure/`, `tachi-privilege-escalation/`, `tachi-repudiation/`) gains a **Pattern Category Disambiguation subsection** drawing the boundary between Cat 1-N pre-existing (signal-class generic surface) and the new mobile-specific Cats. The single-host fallback drops one of the {`privilege-escalation`, `repudiation`} disambiguation subsections.

**Disambiguation contract**: each subsection must clarify (a) the architectural-tell axis that distinguishes pre-existing Cat 1-N from the new mobile Cats, (b) the explicit non-overlap statement (same architecture exhibiting BOTH a pre-existing finding and a new mobile finding does not constitute duplicate emission — the architectural-tells are disjoint), (c) at least one worked example demonstrating the boundary on a hybrid mobile+web architecture.

Specific disambiguation contracts per companion:

- **Spoofing companion** — Cat 1-N pre-existing covers generic spoofing (web session theft, generic credential phishing); new Mobile Cats N+1/N+2 cover mobile-specific credential storage and mobile-specific authn/authz (biometric step-up gaps, mobile session cookie storage in unsecured locations). Disambiguation hinge: the **architectural-tell axis** distinguishes generic credential exposure (any client / server) from mobile-specific credential exposure surfaces (Keychain on iOS / Keystore on Android / SharedPreferences / SQLite credential caches / clipboard credential leakage / biometric prompt bypass paths).
- **Tampering companion** — Cat 1-9 pre-existing covers generic tampering + LLM data-poisoning Cat 8/9/10 (post-F-6 inventory at 221 lines); new Cats 11/12/13 cover mobile-specific supply chain integrity (SDK provenance, dependency injection at install time), mobile IPC input validation (Intent / URL-scheme / ContentProvider gates), and mobile binary protections (root-detection, RASP, code-obfuscation). Disambiguation hinge: Cat 1-9 are signal-class generic; new Cats 11-13 require **mobile topology indicators** (declared mobile platform, IPC surface, mobile binary distribution) — without these indicators, the new Cats emit zero findings per the mobile topology gate.
- **Info-disclosure companion** — Cat 1-N pre-existing covers generic info-disclosure (server-side log leakage, response-body PII); new Cats N+1/N+2/N+3/N+4 cover mobile-specific transport security, mobile privacy controls (FLAG_SECURE / clipboard / pasteboard), mobile data storage (SQLite / Keychain / SharedPreferences), and mobile cryptography (weak KDFs on PIN / custom-rolled crypto). Disambiguation hinge: the four new Cats are layered by attack surface — transport (M5) → privacy controls at runtime (M6) → at-rest storage (M9) → cryptographic primitives underlying both transport and storage (M10). Same architecture may surface findings in multiple new Cats without duplication; each Cat targets a distinct architectural surface.
- **Privilege-escalation companion** — Cat 1-N pre-existing covers generic privilege-escalation (server-side IDOR, role-confusion); new Cat N+1 covers mobile-specific misconfiguration enabling privilege gain (debug ContentProvider exports, missing attestation, root-detection bypass). Disambiguation hinge: server-side IDOR is generic to any client/server architecture; mobile misconfiguration requires inspection of the mobile manifest (`AndroidManifest.xml` exported flags), the mobile app build configuration (debug-flag handling, R8/ProGuard mappings), and the mobile attestation surface (Play Integrity / DeviceCheck integration).
- **Repudiation companion** — Cat 1-N pre-existing covers generic repudiation (server-side audit-log gaps, missing transaction signing); new Cat N+1 covers mobile-specific misconfiguration enabling accountability loss (Log.d/NSLog leakage, disabled Crashlytics, missing tamper-evident timestamping on transaction logs). Disambiguation hinge: server-side audit-log gaps are inspected via server log infrastructure; mobile audit-log gaps are inspected via mobile logging instrumentation (Android Log.* / iOS NSLog / OSLog) and the mobile crash-reporting integration (Firebase Crashlytics / Sentry / Bugsnag). The mobile-specific tells are orthogonal to server-side tells; both may co-exist on a hybrid mobile+server architecture without duplication.

**Worked example — hybrid web+mobile fintech app**: a fintech architecture with both an Android/iOS client (mobile binary distribution; deep-link Intent for transfer-confirmation; biometric step-up; secure-storage paths for refresh tokens) and a server-side admin panel (web session cookies; server-side role-based authorization) may legitimately surface the following findings without duplication:
1. Pre-existing spoofing Cat 1-N finding on the server-side admin panel (web session vulnerability).
2. New spoofing Cat N+1 finding on the mobile client (refresh token in SharedPreferences without encryption).
3. Pre-existing info-disclosure Cat 1-N finding on server-side response-body PII leakage.
4. New info-disclosure Cat N+1 finding on cleartext HTTP residual to a deprecated API endpoint on the mobile client.
5. New tampering Cat 12 finding on the mobile client's deep-link parameter handler (unvalidated `amount` extra in transfer-confirmation Intent).
6. New privilege-escalation Cat N+1 finding on a debug ContentProvider exposed in release builds.

Six findings, six disjoint architectural-tells, zero duplication. The Pattern Category Disambiguation subsection in each companion makes this non-overlap explicit so downstream consumers (`risk-scorer`, `control-analyzer`, `threat-report`) can render them adjacent without reconciliation logic.

**Notes for Wave 2.x execution**: Wave 2.x parallel companion-catalog authoring across 5 streams (or 4 in single-host fallback) requires Pattern Category Disambiguation subsections to be authored as the **last anchor point** in each companion's edit set, after all new Pattern Categories have been appended. The Disambiguation subsection's prose references the new Cat numbers and pre-existing Cat 1-N range — both numbers must be locked before the Disambiguation prose stabilizes. Wave 2.x team-lead checkpoints should verify Disambiguation prose accuracy at each companion's authoring completion before signing off the companion as Wave-2.x-complete. The architect-recommended Wave 2.x ordering: (a) author all new Pattern Categories per companion → (b) verify byte-identity on Cat 1-N pre-existing → (c) author Pattern Category Disambiguation subsection → (d) author Primary Sources extension → (e) team-lead checkpoint. This ordering is structurally symmetric with the F-6 ADR-035 Wave 2.x execution pattern.

Mirrors **F-3 ADR-032 D7 + F-5 ADR-034 D7 + F-6 ADR-035 D-9 precedent** at four-or-five-agent scale. Same architecture (e.g., a hybrid web+mobile fintech app with both a server-side admin panel and an Android/iOS client) may legitimately surface both pre-existing and Mobile findings without duplication.

### Decision 10 — No `source_attribution` populator wiring extension (F-A3 deferral)

F-7 cites OWASP Mobile primaries (M1-M10:2024) via the existing `references:` field per the F-A2 referential-integrity contract (ADR-028 D5). Cross-reference: `schemas/taxonomy/owasp.yaml` was verified at Wave 1.0 T012 to contain all 10 catalog-resolvable M1-M10 entries.

F-7 does **NOT** extend `source_attribution` populator wiring on any of the host agents. The `source_attribution: []` empty-array default per finding-format-shared.md is preserved across all 5 hosts (or 4 in the single-host fallback). That scope belongs to **F-A3** (one-way dependency: F-7 → F-A3). F-7 is the **third BLP-01 detection feature to defer populator wiring** (after F-5 and F-6).

This is structurally symmetric with ADR-034 D8 (F-5 deferred populator wiring) and ADR-035 D-10 (F-6 deferred populator wiring). The three-feature precedent (F-5 + F-6 + F-7) establishes a clear pattern: **enrichment-branch features uniformly defer F-A3 populator wiring**, while net-new agent features (F-1 + F-2 + F-4) ship populator wiring inline. The split simplifies F-A3's eventual scope: F-A3 owns the populator wiring for ALL existing host agents that emit BLP-01 findings via enrichment, while net-new agents continue to ship inline populator wiring at their own genesis.

**F-A3 cumulative scope after F-7**: F-A3 will own populator wiring across (post-F-7) **eight host agents** that have received BLP-01 enrichment without inline populator wiring: F-3 enriched `tool-abuse` (1 host); F-5 enriched `denial-of-service` + `model-theft` (2 hosts); F-6 enriched `tampering` + `data-poisoning` + `model-theft` (3 hosts; `model-theft` overlaps with F-5); F-7 enriches `spoofing` + `tampering` + `info-disclosure` + `privilege-escalation` + `repudiation` (5 hosts in dual-host path; 4 in single-host fallback; `tampering` overlaps with F-6). The unique-host count after F-7 dual-host path: `tool-abuse` + `denial-of-service` + `model-theft` + `tampering` + `data-poisoning` + `spoofing` + `info-disclosure` + `privilege-escalation` + `repudiation` = **9 unique hosts**. F-A3's populator wiring scope is therefore bounded; the cumulative scope is a one-way function of BLP-01 feature execution and stabilizes at F-A3 ship-date.

## Consequences

### Positive

- **OWASP four-framework total reaches 40/40 Covered post-F-7** (LLM Top 10:2025 = 10/10 + Agentic Top 10:2026 = 10/10 + ML Top 10:2023 = 10/10 + Mobile Top 10:2024 = 10/10). Tachi's existing detection agents close the OWASP four-framework family in full. Establishes tachi's **OWASP coverage matrix** as the most comprehensive in the AI security tooling ecosystem.

- **Heuristic A four-or-five-agent enrichment scope established**: F-7 demonstrates the enrichment-branch protocol scales linearly through four-or-five-agent scope without introducing new operational pathologies. Establishes precedent for F-8 Tier 3 web/API refinement bundle and beyond. The **structural symmetry forecast in ADR-035 line 77 ("extensibility to ≥4-agent fan-out implied by structural symmetry") is now operationally validated**.

- **No schema bump, no consumers-list edit, no orchestrator edit**: fourth symmetric application of the no-schema-bump discipline (after F-3, F-5, F-6); preserves the schema versioning timeline for future net-new-agent features at F-8+. The schema-version stability through 4 consecutive enrichment features is a strong signal that the **enrichment branch is the correct structural pattern** for OWASP coverage closure on existing detection-tier signal classes.

- **M8 dual-host carve operationalizes architectural-tell axis** as a new structural pattern alongside ADR-035's entry-decomposition axis (ML06 corpus-vs-artifact). Future BLP-01 features can identify whether a given OWASP entry decomposes via entry-decomposition (distinct attack-surface targets) or architectural-tell (interlocking consequences of one underlying misconfiguration) at PRD time, with corresponding host-agent assignments per axis.

- **M4 cross-agent boundary clarification (D-5)** preserves disjoint detection vocabularies on hybrid LLM+mobile architectures — a category that is expected to grow with on-device LLMs and mobile agentic apps. Sets precedent for future cross-axis boundary decisions when LLM-tier and STRIDE-tier surfaces overlap.

### Negative

- **Build complexity at four-or-five-agent scope** is higher than at three-agent scope. The 5 host edits (or 4 in single-host fallback) span 5 STRIDE companions with 5 distinct Pattern Category Disambiguation subsections. Wave 2.x execution must coordinate across 5 parallel companion-catalog authoring streams; team-lead checkpoint frequency may need to increase from F-6's three-agent precedent.

- **3-of-3 ATT&CK Mobile catalog gap** (T1474/T1626/T1398 prose-only) is the worst-case scale observed in BLP-01 to date (vs F-5's 1-of-1 and F-6's 3-of-6). Future F-A1.1 follow-on must upgrade these to structured `references:` entries to close the catalog gap; until then, the prose-only invariant must be enforced by Wave 4 grep validation on the `mobile-banking-app/` mutation target.

- **Build envelope exposure to dual-host vs single-host fallback decision**: if Triad governance reverses the dual-host recommendation late in the build cycle (e.g., at Wave 2.x), the M8 carve must be re-allocated to a single host with corresponding companion-catalog edits, which would invalidate part of the Wave 2.x parallel authoring streams. Risk mitigation: the dual-host vs single-host decision should be locked at PRD-time / spec-time before Wave 2.x begins; this ADR records the architect-recommended default to inform that lock.

### Neutral

- F-7's extensibility-to-≥4-agent-fan-out fulfillment closes the multi-agent enrichment forecast chain (ADR-032 → ADR-034 → ADR-035 → ADR-036) at four-or-five-agent scope. F-8 and beyond will need to motivate any new structural pattern from the architectural-tell + entry-decomposition + signal-class taxonomy already established; the BLP-01 ADR family now covers the principal structural patterns for enrichment-branch operations.

- **Forward scope marker for F-8** (forward-traceability convention inherited from ADR-035 line 77): F-8 Tier 3 web/API refinement bundle is BLP-01's next planned feature after F-7. F-8 is expected to invoke the enrichment branch at a smaller agent-count scope than F-7 (likely 2-3 host agents covering OWASP Web Top 10 / API Top 10 entries that engage `tampering` / `info-disclosure` / `privilege-escalation` STRIDE signal classes — the exact carve depends on PRD-time signal-class analysis). F-8 is expected to follow the no-schema-bump path per the decision tree in Decision 6 because the engaged signal classes already have established prefixes (S/T/I/E/R). The F-7 four-or-five-agent precedent establishes that **the enrichment branch scales through 5-agent scope** (the current STRIDE roster size); features engaging more than 5 STRIDE host agents would either reach the limit or require new signal-class authoring (out-of-scope for enrichment branch, in-scope for net-new agent branch per Decision 6 decision tree).
