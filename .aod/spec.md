## User Scenarios & Testing *(mandatory)*

### User Story 1 — Human-Trust Communication-Axis Detection on Consumer-Facing AI Agents (Priority: P0)

A security analyst threat-models an architecture where an AI agent emits content directly to end users without architectural mechanisms preventing impersonation, over-persuasion, or undisclosed AI authorship. The analyst wants tachi to flag those components with concrete `TE-{N}` findings citing OWASP ASI09:2026 so the trust-exploitation communication-axis risk surface — which neither output-sanitization (F-1 `output-integrity`), factual-integrity (F-2 `misinformation`), nor agent-autonomy/HITL (`agent-autonomy` autonomy axis) covers — is surfaced before the agent reaches end users in production.

**Why this priority**: This is the primary reason F-4 exists. Without the human-trust communication-axis detection signal, the AI threat surface remains asymmetric post-F-1 / F-2 / F-3: input-side comprehensive (`prompt-injection`), output-sanitization comprehensive (`output-integrity`), factual-integrity comprehensive (`misinformation`), agent-tooling comprehensive (`tool-abuse` enriched), agent-behavior/HITL comprehensive (`agent-autonomy` autonomy axis), human-trust communication-axis silent. P0 because the feature has no MVP without this story — the gap is the feature.

**Independent Test**: Given an architecture containing an AI Process matching consumer-facing AI keywords (e.g., `chatbot`, `assistant`, `advisor`, `customer-facing`, `companion`, `coach`) AND exhibiting at least one human-user-facing emission indicator (outgoing Data Flow to a human-named External Entity like "Customer" / "Patient" / "User" / "Student", OR Process description containing human-user-facing emission keywords), running `/tachi.threat-model` emits ≥1 `TE-{N}` finding with `category: agentic`, OWASP ASI09:2026 in `references`, and a populated `source_attribution` array. Given an architecture with no qualifying Process or with only backend-only AI processing (no human-user trust boundary crossing), the agent emits **zero** findings — no speculation.

**Acceptance Scenarios**:

1. **Given** an architecture that includes an AI Process emitting output to a human user (via outgoing Data Flow to a human-named External Entity OR Process description containing human-user-facing emission keywords) with no declared AI-disclosure mechanism, **when** the orchestrator dispatches `human-trust-exploitation`, **then** a `TE-{N}` finding emits with `category: agentic`, `references` citing `OWASP ASI09:2026`, and `source_attribution` containing `{taxonomy: owasp, id: ASI09, relationship: primary}` plus `{taxonomy: cwe, id: CWE-223, relationship: related}`.
2. **Given** an architecture where the AI agent is authorized to emit authority signals (legal/medical/financial claims, decision recommendations) without declared confidence thresholds or source attestation, **when** `human-trust-exploitation` runs, **then** a distinct `TE-{N}` finding surfaces the authority-claim-without-attestation surface with `source_attribution` carrying ASI09 primary plus CWE-345 related.
3. **Given** an architecture where the AI agent forms persistent user-facing "relationships" (persona memory, long-running dialogue state, named persona, emotional-support framing) without declared manipulation safeguards, **when** `human-trust-exploitation` runs, **then** a finding flags persona-boundary / synthetic-relationship risk with named consumer-protection mitigations and CWE-287 + CWE-290 in `source_attribution`.
4. **Given** an architecture where AI output never crosses a human-user trust boundary (pure backend automation, machine-to-machine pipeline, internal-only batch processing), **when** `human-trust-exploitation` runs, **then** **zero findings** emit — no speculation. The agent's two-part emission gate (AI keyword match AND human-user-facing emission indicator) prevents false positives on AI Processes whose output is purely machine-consumed.
5. **Given** an architecture where all four AI-tier surfaces are present on the same Process (prompt-injection input surface, output-sanitization sink, factual-integrity emission, human-user-facing trust emission), **when** the orchestrator runs, **then** `prompt-injection`, `output-integrity`, `misinformation`, and `human-trust-exploitation` may emit findings on the same component — this is correct behavior, not duplication; the four agents cover disjoint Heuristic A signal classes.

### User Story 2 — Named Human-Trust Mitigation Guidance Specific to ASI09 Communication Axis (Priority: P0)

A developer reading a `TE-{N}` finding needs to address it in their consumer-facing AI agent. The developer wants the `mitigation` field to name specific AI-disclosure, confidence-calibration, refusal-pattern, persona-boundary, or synthetic-relationship-safeguard mechanisms matched to the detected pattern category — distinct from output-sanitization (F-1), factual-grounding (F-2), or HITL/autonomy mitigations (`agent-autonomy` autonomy axis) — so the fix can be applied without inventing a mitigation pattern from scratch or re-deriving from primary OWASP / FTC / FDA / ABA / SEC sources.

**Why this priority**: Without actionable, pattern-specific mitigation text, findings are noise from a developer's perspective — flagged-but-not-fixable. Equal priority to Story 1 because developer usability gates adoption and mirrors F-1 / F-2 US-2 priority envelope. The mitigation-class distinctness is what makes the finding actionable and prevents conflation with adjacent agents' mitigation advice.

**Independent Test**: For each emitted `TE-{N}` finding, the `mitigation` field names at least one specific AI-disclosure, confidence-calibration, refusal-pattern, persona-boundary, or synthetic-relationship-safeguard mechanism matched to the finding's pattern category — undisclosed AI authorship → mandatory AI-generation disclosure banner / pre-conversation AI-disclosure splash / per-message AI-source label / refusal pattern for identity-impersonation challenges; authority-claim emission without attestation → confidence-threshold gate with confidence display / source-attestation requirement with citation rendered to user / refusal pattern for authority claims below confidence threshold / calibrated-confidence layer; persuasive-tone manipulation → uncertainty-disclosure layer / temperature-bounded decoder for high-stakes responses / persuasion-pattern classifier with refusal / mandatory hedging language; persona-boundary violations → persona-memory timeout / identity-impersonation refusal pattern / persona-anchor declaration / persona-prompt validation rejecting non-AI identity assertions; synthetic-relationship exploitation → session-length cap with mandatory break / escalation-to-human path / emotional-support disclosure on first turn / dependency-risk classifier / **vulnerable-population safeguards layer**.

**Acceptance Scenarios**:

1. **Given** a `TE-N` finding for the **undisclosed AI authorship** category, **when** the `mitigation` field is read, **then** it names at least one specific AI-disclosure mechanism — e.g., `mandatory AI-generation disclosure banner on every user-facing turn`, `pre-conversation AI-disclosure splash with explicit consent confirmation`, or `per-message AI-source label`.
2. **Given** a `TE-N` finding for the **authority-claim emission without attestation** category, **when** the `mitigation` field is read, **then** it names `confidence-threshold gate on authority-claim emission with confidence display`, `source-attestation requirement on legal/medical/financial claims with citation rendered to user`, or `refusal pattern for authority claims below confidence threshold`.
3. **Given** a `TE-N` finding for the **persuasive-tone manipulation** category, **when** the `mitigation` field is read, **then** it names `uncertainty-disclosure layer`, `temperature-bounded decoder for high-stakes responses`, or `persuasion-pattern classifier with refusal on detected manipulation tactics`.
4. **Given** a `TE-N` finding for the **persona-boundary violations on long-running dialogues** category, **when** the `mitigation` field is read, **then** it names `persona-memory timeout`, `identity-impersonation refusal pattern`, or `persona-anchor declaration on every conversation`.
5. **Given** a `TE-N` finding for the **synthetic-relationship exploitation** category, **when** the `mitigation` field is read, **then** it names `session-length cap with mandatory break / escalation-to-human path`, `emotional-support disclosure on first turn`, or `dependency-risk classifier`. **CRITICAL**: distinguishes **vulnerable-population deployments** (mental health / eldercare / minors / cognitive-impairment users) from **general-consumer deployments** by naming additional safeguards (escalation-to-human on high-emotional-distress detection, mandatory professional-care referral, parental-consent requirement for minors, session-context expiry).
6. **Given** any `TE-N` finding, **when** its `references` array is inspected, **then** at least one entry is `OWASP ASI09:2026` (primary), and `source_attribution` carries that citation as `relationship: primary`. Supporting references (FTC, FDA, ABA, SEC, state-level chatbot disclosure laws, AARP eldercare advocacy) appear in pattern-catalog prose framed per NFR-6 safe-language patterns; `source_attribution` carries only catalog-resolvable IDs per the F-A2 referential-integrity validator.
7. **Given** the agent's output across pattern categories, **when** threat descriptions are inspected, **then** they distinguish between **authorship-disclosure** (undisclosed AI authorship), **authority-attestation** (claims without confidence/source backing), **persuasion-manipulation** (tone-driven exploitation), **persona-boundary** (identity discipline on long-running dialogues), and **synthetic-relationship** (sustained-engagement exploitation) — the five primary ASI09 communication-axis sub-classes per OWASP ASI09:2026 taxonomy.

### User Story 3 — Trust-Exploitation Communication-Axis as a First-Class Distinct Threat Class with Stable ADR Crosswalk (Priority: P0)

An adopter runs tachi on a consumer-facing AI application (customer-support chatbot, mental-health companion, eldercare coach, financial-advisory assistant, legal-research assistant, clinical decision-support agent, AI tutor) and produces a threat-model report. The adopter wants human-trust-exploitation to be surfaced as a first-class threat class **distinct from `output-integrity`, `misinformation`, AND the autonomy axis of `agent-autonomy`**, with ASI09 communication-axis binding captured in a stable ADR so downstream compliance mapping is stable across runs. The four-way distinction lets the adopter separate "machine-victim sanitization" (output-integrity), "machine-consumer factual-integrity" (misinformation), "agent's own behavior boundaries" (agent-autonomy autonomy axis), and "human-victim communication-axis" (human-trust-exploitation) in the adopter-facing report.

**Why this priority**: Equal priority to Stories 1 and 2 because the distinctness discipline gates both (a) the adopter-facing report coherence on architectures exercising multiple AI-tier surfaces simultaneously and (b) the BLP-01 Coverage Matrix transition from ASI09 (communication axis) Planned → Covered. Without explicit distinctness, the report collapses the four signal classes into "AI findings" prose and the Heuristic A discipline established by F-1 / ADR-030 + F-2 / ADR-031 + F-3 / ADR-032 regresses. Additionally, the Naming Disambiguation discipline (vs. existing `trust_exploitation` agentic_pattern enum value from Feature 142) gates report-rendering integrity — without explicit ADR documentation, the two namespaces could collide in adopter perception.

**Independent Test**: ADR-033 at F-4 merge contains an explicit Heuristic A signal-class analysis resolving four scope boundaries plus the ASI09 sub-scope carve-up: (a) distinct from `output-integrity` (machine-victim sanitization per ADR-030 D1), (b) distinct from `misinformation` (machine-consumer factual-integrity per ADR-031), (c) distinct from `agent-autonomy` autonomy axis (vocabulary-disjoint per ADR-032's enrichment-eligibility test — agent-autonomy covers excessive-autonomy / missing-HITL / goal-drift / cascading-failures; communication axis covers AI-disclosure / authority-claim / persuasion / persona / synthetic-relationship), (d) explicit ASI09 sub-scope carve-up between agent-autonomy autonomy axis (agent's own-behavior boundary) and human-trust-exploitation communication axis (agent's effect on human via communication patterns) — both retain ASI-09 in `owasp_references`, neither agent edits the other. The regenerated example architecture exercising consumer-facing AI emission renders `AG-{N}`, `AGP-{N}`, and `TE-{N}` findings adjacent (not synthesized) in the `category: agentic` section of `threat-report.md`. The `## Naming Disambiguation` section of ADR-033 explicitly contrasts the new `human-trust-exploitation` agent name (hyphen-cased, agent-to-human scope) against the existing `trust_exploitation` agentic_pattern enum value (underscore-cased, agent-to-agent multi-agent topology scope per Feature 142 / `maestro-agentic-patterns-shared.md` R-04).

**Acceptance Scenarios**:

1. **Given** a threat-model report that includes `OI-{N}` (output-integrity), `MI-{N}` (misinformation), `AG-{N}` (agent-autonomy — autonomy axis of ASI09), and `TE-{N}` (human-trust-exploitation — communication axis of ASI09) findings on the same consumer-facing AI Process, **when** the AI-category section of `threat-report.md` is read, **then** the four finding families render in the appropriate sections (`category: llm` for OI/MI; `category: agentic` for AG/TE) **without prose synthesis or collapsing** — each finding's ID prefix preserves its signal-class origin and each finding carries its own `source_attribution` primary (LLM05, LLM09, ASI04/06/08/09[autonomy]/10, ASI09[communication] respectively).
2. **Given** ADR-033 at the time of F-4 merge, **when** its Decision section is read, **then** it explicitly resolves four scope boundaries plus the ASI09 sub-scope carve-up: (a) why human-trust-exploitation is **not** subsumed into `output-integrity` (machine-victim sanitization per ADR-030 D1), (b) why human-trust-exploitation is **not** subsumed into `misinformation` (machine-consumer factual-integrity per ADR-031), (c) why human-trust-exploitation is **not** an enrichment of `agent-autonomy` per the F-3 / ADR-032 enrichment-branch path (vocabulary-disjoint per ADR-032's enrichment-eligibility test), (d) why human-trust-exploitation is **not** subsumed into a hypothetical merge with `agent-autonomy` — the autonomy axis (agent's own behavior boundary) and communication axis (agent's effect on human via communication patterns) are distinct sub-axes of ASI09 with non-overlapping mitigation vocabularies, and the carve-up is documented at the ADR layer (no edit to `agent-autonomy.md` required, preserving SC-009 26-file zero-edit invariant).
3. **Given** ADR-033 §"Naming Disambiguation", **when** the section is read, **then** it explicitly documents (a) the new `human-trust-exploitation` agent name (hyphen-cased file/directory convention) vs. existing `trust_exploitation` value in the `agentic_pattern` enum (`schemas/finding.yaml:162`, underscore-cased schema-enum convention; multi-agent topology pattern per Feature 142 / `maestro-agentic-patterns-shared.md:220-231` R-04), (b) the agent-name lexical convention (hyphen-cased) vs. the schema-enum-value lexical convention (underscore-cased), (c) the `TE` finding-prefix as the threat-class abbreviation (agent-name-agnostic — stands for "Trust Exploitation"), (d) the non-overlapping scope boundary (agent-to-human ASI09 vs. agent-to-agent CSA MAESTRO multi-agent topology).
4. **Given** ADR-033 §"DFD Target Decision", **when** the section is read, **then** it explicitly documents (a) `dfd_targets: [Process]` (Process-only) declaration rationale: F-4 mirrors F-1 / F-2 single-DFD-target precedent, (b) human-user trust boundary captured at indicator level within `detection-patterns.md` rather than at dispatch metadata level, (c) no existing AI-tier or agentic-category agent has declared External Entity as a DFD target (only STRIDE-only precedent exists: `repudiation`, `spoofing`), (d) **architectural commitment**: future AI-tier agents wishing to declare External Entity must update this ADR and document the orchestrator-tier dispatch-rules External Entity inclusion pattern explicitly, (e) deferred External Entity pattern available for future enrichment if/when a second AI-tier agent justifies the cost.

### Edge Cases

- **AI Process with AI-keyword match but no human-user-facing emission indicator**: a component matches a dispatch trigger keyword (e.g., component name contains "assistant" or "agent") but its outgoing Data Flows are purely machine-consumed (downstream backend service, internal queue, API endpoint with no human-facing surface). The two-part emission gate (FR-013) requires **both** an AI-agent keyword match AND a human-user-facing emission indicator structurally present in the component's outgoing Data Flows or description — keyword match alone MUST NOT produce a finding. Matches the zero-speculation discipline established by F-1 / F-2.
- **`persona` keyword anti-indicator (architect Q2 LOW-2 discipline)**: an architecture description uses the word `persona` only in a prompt-engineering context (e.g., "the LLM is configured with a customer-service persona prompt to generate responses for downstream API consumers") with no human-user-facing emission, no chatbot/companion/dialogue context, and no human-named External Entity in the outgoing Data Flow. The pattern catalog's `## Detection Scope` Trigger Keywords section MUST include an explicit **persona anti-indicator** subsection: when `persona` appears WITHOUT human-user-facing emission indicators, agent emits zero findings on that component — preventing the most likely false-positive pattern given the dual-use nature of the `persona` keyword in LLM prompt engineering vs. consumer-facing agents.
- **`agentic-app` baseline regeneration interaction with F-1 OI / F-2 MI / F-3 enriched AG findings**: `agentic-app` was regenerated by F-1 on 2026-04-19 (carries `OI-{N}` findings) and again by F-3 on 2026-04-26 (carries the new AG-8 inter-agent-communication finding from F-3 enrichment). - **Byte-identity baseline coverage on 5 non-consumer-facing examples**: `web-app`, `microservices`, `ascii-web-api`, `mermaid-agentic-app`, `free-text-microservice` have no consumer-facing AI Processes with human-user-facing emission. Plan stage re-verifies this invariant on the post-F-3 6-baseline state.

Mitigations are framed neutrally ("Implement X", "Configure Y", "Enable Z") without persuasive language ("you really should...", "this is critical to fix immediately"). The pattern-catalog prose is the **negative-space example** — by modeling disclosure discipline in its own voice, the agent demonstrates the mitigation pattern it recommends.

## Requirements *(mandatory)*

### Functional Requirements

### Non-Functional Requirements

- **NFR-001 (Determinism)**: The new agent's output MUST be deterministic per ADR-021. Same architecture input → same finding set → same finding ordering. Pattern matching is keyword-based + structural (DFD-element-type filtering + indicator-level human-user-facing emission filtering); no LLM-judgment in pattern selection. Severity computation uses the OWASP 3×3 matrix from `severity-bands-shared.md` deterministically. [Maps to PRD NFR-1]
- **NFR-002 (Backward Compatibility)**: The 5 non-consumer-facing example PDF baselines (`web-app`, `microservices`, `ascii-web-api`, `mermaid-agentic-app`, `free-text-microservice`) regenerate byte-identically under `SOURCE_DATE_EPOCH=1700000000` (SC-006). No qualifying Process + human-user-facing emission indicator in those baselines means zero new findings. [Maps to PRD NFR-2]
- **NFR-003 (Tier-Cap Compliance)**: Agent file ≤150 lines (AI tier cap per ADR-023). Hard ceiling 180 lines. Companion `detection-patterns.md` has no formal line cap but should remain in the ~150-300 line range typical of the existing 14 companion files for readability. [Maps to PRD NFR-3]
- **NFR-004 (Zero New Dependencies)**: Empty diff on `pyproject.toml`, `requirements*.txt`, `package.json` (SC-008). The new agent uses only existing tachi infrastructure (orchestrator dispatch, shared references, schema, parser). [Maps to PRD NFR-4]
- **NFR-005 (Security & Privacy)**: The agent reads architecture descriptions and emits findings. No network calls, no file writes outside the standard finding-emission path, no execution of LLM-output content. The pattern catalog uses anonymized worked examples — no real-world adopter data. [Maps to PRD NFR-5]
- **NFR-006 (Content Sensitivity / Vulnerable-Population Safe-Language Patterns)**: Worked examples cite vulnerable-population deployments (mental health, eldercare, minors, cognitive-impairment users). Catalog prose MUST NOT itself emit content that could be mistaken as authoritative consumer-protection guidance, product endorsement, or legal advice. Examples MUST be clearly-fictional with **four explicit safe-language patterns**: (1) **Hypothetical framing** — all worked-example scenarios prefixed with "Hypothetical: ..." (no implication of real product critique); (2) **Regulatory framing** — all regulatory citations framed as "see, e.g., [regulation] — for context, not legal interpretation" (specifically forbids: legal-interpretation phrasing, mandate-language "X requires Y", or implied-endorsement of specific compliance approaches); (3) **Distress-signal framing** — all distress-signal examples described in **non-clinical terms** (avoid clinical phrasing like "suicidal ideation"; use neutral framing like "user expresses high emotional distress"); (4) **No real institutional names** — no real clinician/lawyer/advisor/educator identities, no real hospital/law-firm/advisory-firm/school names, no real product names that could be misread as authoritative regulatory guidance or competitive critique. Vulnerable-population examples (category 5) MUST cite consumer-protection rationale in prose AND MUST recommend escalation-to-human path as a baseline mitigation. Patterns applied to all 5 pattern categories; vulnerable-population worked examples carry the strictest application. [Maps to PRD NFR-6 — architect Pre-Mortem fix R7]
- **NFR-007 (Disclosure Discipline / Avoid Self-Demonstration)**: The agent's prose, mitigation text, and worked examples MUST themselves model appropriate AI-disclosure discipline. The agent MUST NOT use persuasive language in its mitigation recommendations (no "you really should..." or "this is critical to fix immediately"). Mitigations are framed neutrally with named mechanisms ("Implement X", "Configure Y", "Enable Z") and citation-backed rationale. The pattern-catalog prose is the **negative-space example** — by modeling disclosure discipline in its own voice, the agent demonstrates the mitigation pattern it recommends. [Maps to PRD NFR-7]
