# Feature Specification: `misinformation` Threat Agent (OWASP LLM09:2025)

**Feature Branch**: `206-misinformation-threat-agent`
**Created**: 2026-04-23
**Status**: Draft
**Input**: User description: "PRD: 206 - misinformation-threat-agent"
**PRD**: [docs/product/02_PRD/206-misinformation-threat-agent-2026-04-23.md](../../docs/product/02_PRD/206-misinformation-threat-agent-2026-04-23.md)
**BLP-01 Phase**: Tier 1 — second net-new AI-tier threat detection agent (follows F-1 Feature 201 merged 2026-04-19); closes LLM09:2025 on the Coverage Matrix

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Ungrounded-Factual-Output Detection (Priority: P1)

A security analyst threat-models an architecture where an LLM emits factual, citation-bearing, or decision-critical output without a declared grounding, verification, or confidence-calibration mechanism. The analyst wants tachi to flag those components with concrete `MI-{N}` findings citing OWASP LLM09:2025 so the misinformation risk surface — which neither input-side `prompt-injection` nor output-side `output-integrity` covers — is surfaced before the output reaches end users, downstream automated decisions, or regulatory-exposed systems.

**Why this priority**: This is the primary reason F-2 exists. Without the factual-integrity detection signal, the LLM threat surface remains asymmetric post-F-1: input-side comprehensive (`prompt-injection`), output-sanitization comprehensive (`output-integrity`), factual-integrity silent. P1 because the feature has no MVP without this story.

**Independent Test**: Given an architecture containing an LLM Process whose output exhibits factual-output indicators (factual claims, citations, recommendation/decision outputs, or high-stakes domain signals) without declared grounding, running `/tachi.threat-model` emits ≥1 `MI-{N}` finding with `category: llm`, OWASP LLM09:2025 in `references`, and a populated `source_attribution` array. Given an architecture with no qualifying Process or only purely-stylistic LLM output, the agent emits **zero** findings — no speculation.

**Acceptance Scenarios**:

1. **Given** an architecture that includes an LLM Process producing factual or citation-bearing output with no declared RAG grounding, citation verification, or confidence scoring, **when** the orchestrator dispatches `misinformation`, **then** an `MI-N` finding emits with `category: llm`, `references` citing `OWASP LLM09:2025`, and `source_attribution` containing `{taxonomy: owasp, id: LLM09, relationship: primary}` plus `{taxonomy: cwe, id: CWE-345, relationship: related}`.
2. **Given** an architecture that includes a RAG pipeline but retrieved sources are **not labeled in the output** (the "citation fabrication" pattern category), **when** `misinformation` runs, **then** a distinct `MI-N` finding surfaces the citation-fabrication surface with `source_attribution` carrying LLM09 primary plus CWE-345 related.
3. **Given** an architecture where LLM output drives an automated decision (approve/deny/recommend, triage/classify, content-moderation ruling) with no human-in-the-loop or secondary verification, **when** `misinformation` runs, **then** a finding flags overreliance risk with `source_attribution` carrying LLM09 primary plus CWE-223 related, and a `mitigation` field naming HITL review gate or secondary-verification mechanism.
4. **Given** an architecture where LLM output is **purely stylistic** (copy generation, summarization of user-supplied text with no factual-claim emission), **when** `misinformation` runs, **then** **zero findings** emit — no speculation. The agent's two-part emission gate (LLM keyword match AND factual-output indicator) prevents false positives on stylistic-only LLM Processes.
5. **Given** an architecture where all three LLM-side surfaces are present on the same Process (prompt-injection input surface, output-sanitization sink, factual-output emission), **when** the orchestrator runs, **then** `prompt-injection`, `output-integrity`, and `misinformation` emit findings on the same component — this is correct behavior, not duplication; the three agents cover disjoint Heuristic A signal classes.

### User Story 2 — Named Grounding / Verification Mitigation Guidance (Priority: P1)

A developer reading an `MI-{N}` finding needs to address it in their application. The developer wants the `mitigation` field to name specific grounding, verification, HITL, or calibration mechanisms matched to the detected pattern category so the fix can be applied without inventing a mitigation pattern from scratch or re-deriving from primary OWASP / NIST / ATLAS sources.

**Why this priority**: Without actionable, pattern-specific mitigation text, findings are noise from a developer's perspective — flagged-but-not-fixable. Equal priority to Story 1 because developer usability gates adoption and mirrors F-1's US-2 priority envelope.

**Independent Test**: For each emitted `MI-{N}` finding, the `mitigation` field names at least one specific grounding, verification, HITL, or calibration mechanism matched to the finding's pattern category — ungrounded factual emission → mandatory RAG grounding with per-claim source attribution / retrieval-strength metric declaration / confidence-calibration layer; citation fabrication → output-time citation verification against retrieved source URIs / strict citation-token decoder constraint; overreliance / missing HITL → human-in-the-loop review queue / risk-threshold-based auto-escalation / AI-provenance disclosure; retrieval-grounding gaps → retrieval-strength metric (hit-rate, recall@k) / per-query retrieval-score threshold / versioned retrieval corpus with staleness policy; confidence-calibration absence → calibration layer (temperature scaling + ECE monitor) / refusal pattern for low-confidence queries / calibrated-confidence exposure on output.

**Acceptance Scenarios**:

1. **Given** an `MI-N` finding for the **ungrounded factual emission** category, **when** the `mitigation` field is read, **then** it names at least one specific grounding mechanism — e.g., `mandatory RAG grounding with per-claim source attribution`, `retrieval-strength metric declaration (hit-rate, recall@k)`, or `confidence-calibration layer (temperature scaling + ECE monitor)`.
2. **Given** an `MI-N` finding for the **citation fabrication** category, **when** the `mitigation` field is read, **then** it names `output-time citation verification against retrieved source URIs`, `strict citation-token constraint on decoder output`, or equivalent concrete check.
3. **Given** an `MI-N` finding for the **overreliance / missing HITL** category, **when** the `mitigation` field is read, **then** it names `human-in-the-loop review queue on decision-critical output` and distinguishes **consumer-facing high-stakes** (HIGH or CRITICAL risk level per OWASP 3×3) from **internal low-stakes** (MEDIUM or below).
4. **Given** any `MI-N` finding, **when** its `references` array is inspected, **then** at least one entry is `OWASP LLM09:2025` (primary), and `source_attribution` carries that citation as `relationship: primary`. Supporting references (`MITRE ATLAS AML.T0042 Verify Attack`, `NIST AI 600-1 §2.4 Hallucination`) appear in prose inside the pattern catalog's Primary Sources list; `source_attribution` carries only catalog-resolvable IDs per the F-A2 referential-integrity validator.
5. **Given** the agent's output across pattern categories, **when** threat descriptions are inspected, **then** they distinguish between **factual-emission** (ungrounded output with no grounding declared), **citation-integrity** (RAG present but sources not labeled), and **decision-overreliance** (auto-action without HITL) — the three primary LLM09 sub-classes per OWASP LLM09:2025 taxonomy.

### User Story 3 — Factual-Integrity as a First-Class Distinct Threat Class (Priority: P1)

An adopter runs tachi on an AI-assisted medical, legal, financial, or decision-critical application. The adopter wants misinformation to be surfaced as a first-class threat class **distinct from both `prompt-injection` and `output-integrity`** so the adopter-facing report separates "attacker-induced wrong output" (prompt-injection), "output crosses unsanitized boundary into execution sink" (output-integrity), and "ungrounded wrong output" (misinformation). ADR-031 records the Heuristic A signal-class rationale cross-referencing ADR-030's Decision 1 scope boundary, so the distinctness is documented in the public ADR record rather than carried implicitly in agent prose.

**Why this priority**: Equal priority to Stories 1 and 2 because the distinctness discipline gates both (a) the adopter-facing report coherence on architectures exercising multiple LLM-side surfaces simultaneously and (b) the BLP-01 Coverage Matrix transition from LLM09 Planned → Covered. Without explicit distinctness, the report collapses the three signal classes into "LLM findings" prose and the Heuristic A discipline established by F-1 / ADR-030 regresses.

**Independent Test**: ADR-031 at F-2 merge contains an explicit Heuristic A signal-class analysis resolving three scope boundaries: (a) distinct from `prompt-injection` (input-side attacker-controlled injection — machine-attacker, input primitives), (b) distinct from `output-integrity` (output-sanitization per ADR-030 Decision 1 — machine-victim, bytes/strings/syntax primitives), (c) scoped to factual-integrity (grounding, verification, HITL, calibration — human-victim and decision-cascade-victim, factual-content primitives). The regenerated example architecture exercising all three surfaces renders `LLM-{N}`, `OI-{N}`, and `MI-{N}` findings adjacent (not synthesized) in the `category: llm` section of `threat-report.md`.

**Acceptance Scenarios**:

1. **Given** a threat-model report that includes `LLM-{N}` (prompt-injection), `OI-{N}` (output-integrity), and `MI-{N}` (misinformation) findings on the same LLM Process, **when** the AI-category section of `threat-report.md` is read, **then** the three finding families render in the same `category: llm` section **without prose synthesis or collapsing** — each finding's ID prefix preserves its signal-class origin and each finding carries its own `source_attribution` primary (LLM01, LLM05, LLM09 respectively).
2. **Given** ADR-031 at the time of F-2 merge, **when** its Decision section is read, **then** it explicitly resolves three scope boundaries: (a) why misinformation is **not** subsumed into `prompt-injection` (input-side injection is a distinct signal class — attacker-controlled input vs. architecture-level grounding absence), (b) why misinformation is **not** subsumed into `output-integrity` (F-1's scope is downstream-execution-sanitization per ADR-030 Decision 1 — machine-victim, bytes/strings/syntax primitives; F-2's scope is factual-integrity — factual-content primitives), (c) why misinformation is **not** deferred to a post-Tier-1 feature.
3. **Given** a regenerated example architecture that exercises the factual-output flow, **when** `/tachi.threat-model` is run, **then** the `threats.md` AI-category rendering groups `MI-{N}` findings adjacent to (not synthesized with) `LLM-{N}` and `OI-{N}` findings — the grouping preserves the three-signal-class discipline.
4. **Given** the BLP-01 Coverage Matrix post-F-2 merge, **when** LLM09:2025 row is inspected, **then** it transitions **Planned → Covered** with F-2 (Feature 206) named as the closure feature.
5. **Given** ADR-031 as Proposed by Day 1 Wave 1.1 (per PRD Q5 leaning), **when** Day 1 Wave 1.2 pattern catalog authoring begins, **then** the signal-class discipline is committed and downstream authoring proceeds without architectural-scope rework. Deviation from this sequencing requires explicit architect override at plan time.

### Edge Cases

- **LLM Process with LLM keyword match but no factual-output indicator**: a component matches a dispatch trigger keyword (e.g., component name contains "LLM") but its output is purely stylistic — copy generation, summarization of user-supplied text, translation. The two-part emission gate (FR-011) requires **both** an LLM keyword match AND a factual-output indicator structurally present in the component's description or connected Data Flows — keyword match alone MUST NOT produce a finding. Matches the zero-speculation discipline established by F-1.
- **`agentic-app` baseline regeneration interaction with F-1 OI findings**: `agentic-app` was regenerated by F-1 on 2026-04-19 and now carries `OI-{N}` findings. If F-2 selects `agentic-app` as its regeneration target (Q4 PM leaning: extend with factual-output sub-component), the expected-diff is additive — new `MI-{N}` findings on top of existing `LLM-{N}` + `OI-{N}` findings. Plan stage audits the expected-diff surface to distinguish intended additive changes from regression signals.
- **F-A2 referential validation failure on absent catalog IDs**: if an `MI-{N}` finding cites a taxonomy ID not present in `schemas/taxonomy/{owasp,cwe,mitre-atlas,nist-ai-rmf}.yaml`, `validate_source_attribution` rejects the finding at parse time. Pattern catalog worked examples MUST cite only IDs verified present — `AML.T0042` is confirmed absent and remains prose-only per PRD FR-5 verification.
- **`AML.T0042 Verify Attack` prose-only citation**: PRD-time audit confirms `schemas/taxonomy/mitre-atlas.yaml` carries 12 AML techniques (`T0010, T0018, T0020, T0024, T0051, T0054, T0057, T0058, T0059, T0060, T0061, T0062`); `T0042` is absent. Pattern-catalog prose retains the `AML.T0042 Verify Attack` citation for adversarial-grounding context; `source_attribution` anchors only on LLM09 + applicable CWE IDs. If a future feature populates the catalog with AML.T0042, the pattern-catalog citation upgrades from prose-only to `source_attribution` entry as a separate additive change.
- **`NIST AI 600-1 §2.4 Hallucination` prose-only citation**: section-level IDs are not populated in `schemas/taxonomy/nist-ai-rmf.yaml` at PRD time. The citation remains prose-only in the pattern catalog's Primary Sources list; `source_attribution` does not cite NIST section-level IDs until a future catalog population feature lands.
- **Three-agent simultaneous emission on a single LLM Process**: when an architecture exercises prompt-injection input surface + output-sanitization sink + factual-output emission on the same component, all three agents fire. This is **correct behavior**, not duplication — the three agents cover disjoint Heuristic A signal classes (input-injection, output-sanitization, factual-integrity). The report-tier rendering MUST preserve the three ID-prefix families separately without synthesis.
- **Orchestrator dispatch-list drift during F-1 carry-over reconciliation**: three F-1 carry-over callsites (`dispatch-rules.md:120` row, `orchestrator.md:296` sequential-mode text, `orchestrator.md:370` LLM Threats row) currently reference the pre-F-1 three-agent LLM list. F-2 reconciles these to the five-agent state (adds both F-1's `output-integrity` and F-2's `misinformation` in the same edit). Grep-check at Day 2 AM confirms the quintet is consistent across every callsite.
- **Schema bump collision with concurrent BLP-01 Tier 1 features**: F-2 ships solo in the 2026-04-27 → 2026-04-29 window (verified at PRD time: no F-3/F-4/F-5 Issues at stage:define or later, no competing PRs). If F-3 or F-5 enter build concurrently, multi-feature scheduling requires sequential discipline on the 4 additive-edit surfaces (`schemas/finding.yaml` schema bump, `dispatch-rules.md` LLM list + table row, `finding-format-shared.md` consumers list, `orchestrator.md` dispatch list + sequential-mode + LLM Threats row). Team-lead enforces serialization at plan time if the calendar shifts.
- **Byte-identity baseline coverage on the 5 non-factual examples**: `web-app`, `microservices`, `ascii-web-api`, `mermaid-agentic-app`, `free-text-microservice` have no factual-output components. The two-part emission gate (FR-011) guarantees zero `MI-{N}` findings on these, which guarantees unchanged `threats.md` / `risk-scores.md` / `compensating-controls.md` content, which guarantees byte-identical PDFs under `SOURCE_DATE_EPOCH=1700000000`. Plan stage re-verifies this invariant on the pre-F-2 6-baseline state in case F-1's `agentic-app` regeneration introduced subtle dispatch reordering.
- **Worked-example content sensitivity in healthcare / legal / finance domains**: pattern-catalog worked examples cite high-stakes domains where careless prose could itself emit misinformation. NFR-6 mandates clearly-fictional framing ("a hypothetical clinical-decision-support system...", "a generic legal-research tool...", "a synthetic financial-advisory component..."); no real institutional names, no real clinician / lawyer / advisor identities, no real regulatory-citation examples. Senior-backend-engineer + code-reviewer double-check at Wave 2.2.

### Key Entities

- **`misinformation` agent** (`.claude/agents/tachi/misinformation.md`) — new lean AI-tier threat detection agent. Inputs: DFD architecture description. Outputs: ordered list of `MI-{N}` findings per F-A2 schema.
- **`tachi-misinformation` skill** (`.claude/skills/tachi-misinformation/`) — companion skill directory hosting the pattern catalog. Contains `README.md` + `references/detection-patterns.md`.
