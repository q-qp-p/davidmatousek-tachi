# Typst Template Contract — Variable Bindings

Domain reference for the tachi report-assembler agent. Defines the complete interface contract between `report-data.typ` (generated by the agent/extraction script) and the Typst template system. All variable names, types, and structures must match exactly.

---

## Architecture

The Typst template system uses a data injection pattern:

- `main.typ` -- master orchestrator that imports `report-data.typ` and conditionally includes page templates
- `report-data.typ` -- runtime-generated data file containing all extracted variables as Typst `#let` bindings
- `shared.typ` -- shared design tokens (colors, fonts, page geometry)
- Page templates (`cover.typ`, `executive-summary.typ`, etc.) -- each renders one page type

The agent generates `report-data.typ` in `templates/tachi/security-report/`. The extraction script writes it to the same location.

---

## File Header

Every generated `report-data.typ` must begin with:

```typst
// =============================================================================
// Report Data: Auto-generated by tachi report-assembler agent
// =============================================================================
// This file is generated at runtime and imported by main.typ.
// Do NOT edit manually — it will be overwritten on next generation.
// =============================================================================
```

## Variable Definitions

### Metadata Variables

```typst
// --- Metadata ----------------------------------------------------------------
#let project-name = "{project_name}"
#let assessment-date = "{assessment_date}"
#let classification = {classification_value}  // string or none
```

- `project-name`: Extracted from threats.md project name or title override
- `assessment-date`: ISO 8601 date string (YYYY-MM-DD)
- `classification`: Either `"{value}"` (quoted string) or `none` (Typst keyword, no quotes)

### Severity Count Variables

```typst
// --- Severity Counts ---------------------------------------------------------
#let critical-count = {N}
#let high-count = {N}
#let medium-count = {N}
#let low-count = {N}
#let total-findings = {N}
```

All values are integers (unquoted). If risk-scores.md has different counts from threats.md, prefer the risk-scores.md counts.

### Page Inclusion Flags

```typst
// --- Page Inclusion Flags ----------------------------------------------------
#let has-threat-report = {true/false}
#let has-risk-scores = {true/false}
#let has-compensating-controls = {true/false}
#let has-funnel-image = {true/false}
#let has-baseball-image = {true/false}
#let has-architecture-image = {true/false}
```

All values are Typst booleans (unquoted `true` or `false`).

### Data Source Tier

```typst
// --- Data Source Tier ---------------------------------------------------------
#let data-source-tier = {1/2/3}
```

Integer value matching the selected tier from artifact detection.

### Image Paths

```typst
// --- Image Paths (relative to templates/tachi/security-report/) --------------------
#let funnel-image-path = "../../{target_dir}/threat-risk-funnel.jpg"
#let baseball-image-path = "../../{target_dir}/threat-baseball-card.jpg"
#let architecture-image-path = "../../{target_dir}/threat-system-architecture.jpg"
```

**Path resolution**: Typst resolves `#image()` paths relative to the `.typ` file that calls it (which is `full-bleed.typ` in `templates/tachi/security-report/`), NOT relative to the `--root` flag. Use `../../` to navigate from the template directory back to the project root, then append the path to the target directory.

If an image does not exist, set the path to an empty string `""`. The `#if has-*-image` guards in `main.typ` prevent these from being used.

### Executive Narrative

```typst
// --- Executive Narrative -----------------------------------------------------
#let executive-narrative = {narrative_value}
```

Where `narrative_value` is either:
- A string with escaped newlines: `"The assessment identified...\n\nThe system presents..."`
- `none` (Typst keyword) if no narrative available

For multi-line strings, replace literal newlines with `\n`. Maximum 2000 characters; if longer, truncate and append `"..."`.

### Component Distribution

```typst
// --- Component Distribution --------------------------------------------------
#let component-distribution = (
  ("LLM Agent Orchestrator", 11),
  ("MCP Tool Server", 8),
  // ... one tuple per component
)
```

Array of tuples: `(component_name: string, finding_count: integer)`. Set to `none` if no component data available.

### Findings Array

Keys vary by selected tier:

**Tier 1 (Compensating Controls)**:
```typst
#let findings = (
  (id: "S-3", component: "LLM Agent Orchestrator", threat: "Attacker forges...", residual_score: "5.2", residual_severity: "Medium", control_status: "Found", recommendation: "Implement mutual TLS..."),
)
```

**Tier 2 (Risk Scores)**:
```typst
#let findings = (
  (id: "LLM-1", component: "LLM Agent Orchestrator", threat: "Adversarial prompts...", composite_score: "8.3", severity: "High", cvss: "9.8", exploitability: "8.8"),
)
```

**Tier 3 (Threats Only)**:
```typst
#let findings = (
  (id: "S-1", component: "User", threat: "Attacker impersonates...", likelihood: "HIGH", impact: "HIGH", risk_level: "Critical", mitigation: "Implement MFA..."),
)
```

**Important**: All dictionary values must be **strings** (quoted) for consistent rendering. Even numeric scores: `composite_score: "8.3"` not `composite_score: 8.3`.

**String escaping**: Escape double quotes within finding text (`"` becomes `\"`). Escape backslashes (`\` becomes `\\`).

### Coverage Matrix (Tier 1 Only)

```typst
// --- Coverage Matrix ---------------------------------------------------------
#let coverage-matrix = (
  (category: "Spoofing", found: 2, partial: 1, missing: 0),
  (category: "Tampering", found: 1, partial: 0, missing: 1),
  // ... one entry per STRIDE/AI category
)
```

If `has_compensating_controls` is false, set to empty array `()`.

Category values derived from finding ID prefix: S- = Spoofing, T- = Tampering, R- = Repudiation, I- = Information Disclosure, D- = Denial of Service, E- = Elevation of Privilege, AG- = Agentic, LLM- = LLM Threats.

### Controls Array (Tier 1 Only)

```typst
// --- Detailed Controls -------------------------------------------------------
#let controls = (
  (component: "API Gateway", category: "Authentication", status: "Found", evidence: "src/auth/jwt.ts:42", effectiveness: "Strong"),
)
```

If no controls data, set to empty array `()`.

### Coverage Summary (Tier 1 Only)

```typst
// --- Coverage Summary --------------------------------------------------------
#let coverage-summary = (
  total-found: 5,
  total-partial: 3,
  total-missing: 2,
)
```

If no summary data, set to `(total-found: 0, total-partial: 0, total-missing: 0)`.

### Remediation Actions

```typst
// --- Remediation Actions -----------------------------------------------------
#let remediation-actions = (
  (severity: "Critical", finding-id: "S-3", finding-name: "Service Impersonation", recommendation: "Implement mTLS with certificate pinning", sla: "7 days", status: "Not Started"),
)
```

Set to `none` if no remediation data from either source.

**SLA derivation from severity** (when using compensating-controls.md): Critical = "7 days", High = "14 days", Medium = "30 days", Low = "90 days".

### Scope Data Variables

```typst
// --- Scope Data (from threats.md Sections 1-2) ------------------------------
#let scope-components = (
  (name: "API Gateway", type: "Process", description: "Routes API requests"),
)

#let scope-data-flows = (
  (source: "Client", destination: "API Gateway", data: "API Requests", protocol: "HTTPS"),
)

#let scope-trust-boundaries = (
  (zone: "Public Internet", trust-level: "Untrusted", components: "Client App"),
)

#let scope-boundary-crossings = (
  (crossing: "Internet → DMZ", from-zone: "Public", to-zone: "DMZ", components: "API Gateway", controls: "WAF, TLS"),
)

#let scope-component-count = {N}
#let scope-data-flow-count = {N}
#let scope-trust-boundary-count = {N}
```

If no scope data extracted, set arrays to empty `()` and counts to `0`.

### Brand Asset Variables

```typst
// --- Brand Assets -----------------------------------------------------------
#let has-logo-primary = {true/false}
#let has-logo-horizontal = {true/false}
#let logo-primary-path = {path_or_none}
#let logo-primary-dark-path = {path_or_none}
#let logo-horizontal-path = {path_or_none}
```

Paths use the `../../brand/final/` relative path pattern, or `none` if not found. See `brand-asset-guidelines.md` for detection and resolution details.

### Page Visibility Flags

```typst
// --- Page Visibility (defaults, overridden by report-config.typ) ------------
#let show-disclaimer = true
#let show-methodology = true
```

These are fallback defaults. The user's `report-config.typ` takes precedence when present.

### MAESTRO Data Variables

```typst
// --- MAESTRO Data -----------------------------------------------------------
#let has-maestro-data = {true/false}
#let maestro-layer-distribution = (
  (layer-id: "L1", layer-name: "Foundation Model", finding-count: 3, highest-severity: "High"),
  (layer-id: "L2", layer-name: "Data Operations", finding-count: 5, highest-severity: "Critical"),
  // ... one entry per MAESTRO layer with findings
)
#let most-exposed-layer = "L2 — Data Operations"
#let maestro-findings-by-layer = (
  (layer-id: "L1", layer-name: "Foundation Model", findings: (
    (id: "LLM-1", component: "LLM Engine", severity: "High", threat: "Adversarial prompt injection..."),
    (id: "LLM-3", component: "LLM Engine", severity: "Medium", threat: "Model extraction via..."),
  )),
  (layer-id: "L2", layer-name: "Data Operations", findings: (
    (id: "T-2", component: "Vector Store", severity: "Critical", threat: "Data poisoning of..."),
  )),
  // ... one entry per layer group containing findings
)
#let has-maestro-stack-image = {true/false}
#let maestro-stack-image-path = "../../{target_dir}/threat-maestro-stack.jpg"
#let has-maestro-heatmap-image = {true/false}
#let maestro-heatmap-image-path = "../../{target_dir}/threat-maestro-heatmap.jpg"
```

| Variable | Type | Source | Default | Description |
|----------|------|--------|---------|-------------|
| `has-maestro-data` | boolean | threats.md MAESTRO presence | `false` | Gates MAESTRO findings page rendering |
| `maestro-layer-distribution` | array of dicts | Section 6 "Risk by MAESTRO Layer" table | `()` | Per-layer aggregate: `layer-id` (str), `layer-name` (str), `finding-count` (int), `highest-severity` (str) |
| `most-exposed-layer` | string | Computed from layer distribution | `""` | Layer with highest finding count, formatted as "L2 --- Data Operations" |
| `maestro-findings-by-layer` | array of layer groups | Section 3/4 agent tables grouped by layer | `()` | Each group: `layer-id` (str), `layer-name` (str), `findings` (array of finding dicts). Each finding: `id` (str), `component` (str), `severity` (str), `threat` (str) |
| `has-maestro-stack-image` | boolean | File detection | `false` | Gates MAESTRO stack infographic page |
| `maestro-stack-image-path` | string | File path relative to template dir | `""` | Path to `threat-maestro-stack.jpg` |
| `has-maestro-heatmap-image` | boolean | File detection | `false` | Gates MAESTRO heatmap infographic page |
| `maestro-heatmap-image-path` | string | File path relative to template dir | `""` | Path to `threat-maestro-heatmap.jpg` |

**Path resolution**: Same as existing image paths --- relative from `templates/tachi/security-report/` to the target directory using `../../{target_dir}/` prefix.

**Backward compatibility**: `main.typ` provides fallback defaults for all 8 variables. If `report-data.typ` is generated by an older extraction script that omits MAESTRO variables, the defaults gate all MAESTRO pages off (`false` / empty arrays).

## Attack Tree Data Variables (Feature 112)

Two new variables for attack path page generation. Gated by `has-attack-trees` boolean for backward compatibility.

```typst
// --- Attack Tree Data --------------------------------------------------------
#let has-attack-trees = {true/false}
#let attack-trees = (
  (id: "E-3", component: "MCP Tool Server", severity: "Critical",
   title: "Administrative Tool Access via Parameter Manipulation",
   image-path: "../../{target_dir}/attack-trees/E-3-attack-tree.png",
   has-image: true,
   mermaid-text: "flowchart TD\n    ...",
   narrative: "An attacker targets MCP Tool Server by...",
   remediation: ("Implement RBAC validation.", "Add input sanitization.")),
  // ... one entry per Critical/High finding with attack tree
)
```

| Variable | Type | Source | Default | Description |
|----------|------|--------|---------|-------------|
| `has-attack-trees` | `boolean` | `attack-trees/` dir or `threat-report.md` Section 5 | `false` | Gates Attack Path Analysis section |
| `attack-trees` | `array` of dicts | `attack-trees/*.md` files cross-referenced with findings | `()` | One entry per Critical/High finding |

**Entry dict keys**: `id` (string), `component` (string), `severity` (string), `title` (string), `image-path` (string, relative path to PNG or empty), `has-image` (boolean), `mermaid-text` (string, escaped Mermaid source), `narrative` (string, 2-4 sentences), `remediation` (array of strings).

**Image path resolution**: Same as existing image paths --- relative from `templates/tachi/security-report/` to `{target_dir}/attack-trees/{id}-attack-tree.png`. When `mmdc` is unavailable, `has-image` is `false` and the Typst template renders `mermaid-text` as preformatted fallback.

**Backward compatibility**: `main.typ` provides fallback defaults. Older `report-data.typ` files without attack tree variables will have `has-attack-trees = false` and `attack-trees = ()`.

## Report Configuration

The `report-config.typ` file controls page visibility and custom text:

- `show-disclaimer = true` -- default, can be overridden by user
- `show-methodology = true` -- default, can be overridden by user
- `custom-disclaimer-text = none` -- optional custom text
- `custom-footer-text = none` -- optional custom text

The user's config file is imported by `main.typ` and takes precedence. The `show-disclaimer` and `show-methodology` values in `report-data.typ` serve as fallback defaults.

## MAESTRO Data Variables (Feature 091)

Eight new variables added by Feature 091 for MAESTRO layer visualization. All gated by `has-maestro-data` boolean for backward compatibility with pre-Feature 084 output.

| Variable | Type | Source | Default | Description |
|----------|------|--------|---------|-------------|
| `has-maestro-data` | `boolean` | threats.md MAESTRO presence | `false` | Gates MAESTRO findings page |
| `maestro-layer-distribution` | `array` of dicts | Section 6 "Risk by MAESTRO Layer" | `()` | Per-layer aggregate data |
| `most-exposed-layer` | `string` | Computed from layer distribution | `""` | Layer with highest finding count |
| `maestro-findings-by-layer` | `array` of layer groups | Section 3/4 grouped by layer | `()` | Findings grouped by MAESTRO layer |
| `has-maestro-stack-image` | `boolean` | File detection | `false` | Gates stack infographic page |
| `maestro-stack-image-path` | `string` | File path | `""` | Relative path to stack image |
| `has-maestro-heatmap-image` | `boolean` | File detection | `false` | Gates heatmap infographic page |
| `maestro-heatmap-image-path` | `string` | File path | `""` | Relative path to heatmap image |

### maestro-layer-distribution structure

```typst
#let maestro-layer-distribution = (
  (layer-id: "L1", layer-name: "Foundation Model", finding-count: 8, highest-severity: "Critical"),
  (layer-id: "L3", layer-name: "Agent Framework", finding-count: 5, highest-severity: "High"),
)
```

### maestro-findings-by-layer structure

```typst
#let maestro-findings-by-layer = (
  (
    layer-id: "L1",
    layer-name: "Foundation Model",
    findings: (
      (id: "S-1", component: "LLM Engine", severity: "Critical", threat: "Attacker impersonates..."),
      (id: "LLM-3", component: "LLM Engine", severity: "High", threat: "Adversarial prompt..."),
    ),
  ),
)
```

### Image path convention

MAESTRO image file names follow the existing `threat-{template-name}.jpg` pattern:
- `threat-maestro-stack.jpg` — MAESTRO layer stack diagram
- `threat-maestro-heatmap.jpg` — MAESTRO component-layer heatmap

Paths are relative from the Typst template directory: `../../{target_dir}/threat-maestro-stack.jpg`
